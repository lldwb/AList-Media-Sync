- Q: 会话空闲超时如何判定——什么行为算作"操作"可重置 30 分钟倒计时？ → A: 双重保障机制——前端监听用户鼠标/键盘交互（点击、移动、按键等），任意交互重置 30 分钟倒计时，超时后前端主动清除凭据并跳转登录页；同时后端在 30 分钟内未收到该用户任何 API 请求时返回 401，作为服务端兜底保障。

## 用户场景与测试 *（强制）*

### 用户故事 1 — 管理员认证登录（优先级：P1）

作为一名系统管理员，我希望通过浏览器访问 Web 管理界面时输入用户名和密码登录，以便安全地管理系统，防止未授权访问。

**为什么是此优先级**：认证是管理界面的安全入口。没有认证，所有管理功能都会暴露给未授权用户。登录是用户使用管理界面时的第一个交互步骤。

**独立测试**：可以通过浏览器访问管理页面、输入正确的用户名密码后成功进入主页、输入错误凭据后看到错误提示来完整验证。

**验收场景**：

1. **假设** 用户在配置文件中设置了用户名和密码，**当** 用户访问 Web 管理界面，**则** 系统显示登录页面，要求输入用户名和密码。
2. **假设** 用户尚未登录，**当** 用户直接访问任何管理页面路径（如 /sync-tasks），**则** 系统重定向到登录页面。
3. **假设** 用户输入正确的用户名和密码，**当** 点击登录按钮，**则** 系统验证通过，跳转到管理主页，所有后续 API 请求自动携带认证信息。
4. **假设** 用户输入错误的密码，**当** 点击登录按钮，**则** 系统显示"用户名或密码错误"提示，登录页面不清空已输入的用户名。
5. **假设** 用户已登录且在 30 分钟内无操作，**当** 再次访问时，**则** 系统要求重新登录。

---

### 用户故事 2 — 存储引擎管理（优先级：P1）

作为一名系统管理员，我希望在 Web 界面中管理 AList 存储引擎（添加、编辑、删除、测试连接），以便直观地配置用于同步的源和目标存储，无需通过 API 工具操作。

**为什么是此优先级**：存储引擎是所有同步任务的基础。没有存储引擎配置，无法创建任何同步或转码任务。Web 界面必须首先支持此核心资源的可视化管理。

**独立测试**：可以通过登录管理界面后在"存储引擎"页面添加新的 AList 存储配置、测试连接、编辑和删除引擎来完整验证。

**验收场景**：

1. **假设** 用户已登录并进入存储引擎管理页面，**当** 点击"添加引擎"按钮，**则** 弹出表单要求填写引擎名称、API 地址、用户名和密码/Token。
2. **假设** 用户填写了完整的引擎信息，**当** 点击"保存"，**则** 新引擎出现在列表中，凭据字段在列表中以脱敏形式显示（如 `***`）。
3. **假设** 引擎列表中已有已保存的引擎，**当** 用户点击某个引擎的"测试连接"按钮，**则** 系统发送测试请求并在 5 秒内显示连接成功或失败的反馈。
4. **假设** 用户需要修改引擎信息，**当** 点击"编辑"按钮并修改字段后保存，**则** 列表中的信息更新，修改成功有明确提示。
5. **假设** 用户确认删除某个未被任何任务引用的引擎，**当** 点击"删除"按钮并在确认对话框中确认，**则** 引擎从列表中移除。
6. **假设** 某个引擎正被同步任务或 Webhook 规则引用，**当** 用户尝试删除，**则** 系统显示错误提示"该引擎正被使用，无法删除"，并列出引用该引擎的任务名称。

### 用户故事 3 — 同步任务全生命周期管理（优先级：P1）

作为一名系统管理员，我希望在 Web 界面中创建、编辑、启用/禁用、手动触发同步任务，并查看执行历史和进度，以便完整管理文件同步流程，无需使用命令行工具。

**为什么是此优先级**：同步任务是系统的核心功能，用户使用频率最高。完整的可视化管理（创建→监控→历史查看）是 Web 管理界面的主要价值所在。

**独立测试**：可以通过创建同步任务、配置源/目标引擎和路径、触发执行、查看进度变化、查看历史记录来完整验证。

**验收场景**：

1. **假设** 用户已配置了两个存储引擎并进入同步任务管理页面，**当** 点击"创建任务"，**则** 显示表单包含：任务名称、源引擎（下拉选择已有引擎）、目标引擎（下拉选择已有引擎）、源路径、目标路径、同步模式（仅新增/全同步/移动）、执行计划（间隔执行/cron 表达式/手动触发）、文件排除规则（可选）、冲突策略（可选）、"同步后转 MP3"开关。
2. **假设** 用户选择了"间隔执行"模式，**当** 输入间隔秒数，**则** 系统保存间隔配置，任务按间隔自动执行。
3. **假设** 用户选择了"cron 表达式"模式，**当** 输入了 `0 */6 * * *`（每 6 小时），**则** 输入框实时显示下一次执行时间的预览。
4. **假设** 同步任务正在执行中，**当** 用户在任务详情页查看，**则** 显示实时进度：已完成文件数/总文件数、当前处理文件、同步速度、预估剩余时间。
5. **假设** 用户点击某任务的"立即执行"按钮，**当** 该任务上一次执行仍在运行中，**则** 系统弹出提示"任务正在执行中，请稍后再试"。
6. **假设** 用户查看某任务的执行历史，**当** 点击"历史记录"标签，**则** 以分页表格显示每次执行的时间、状态（成功/失败/部分成功/中断）、处理文件数、成功数、失败数，失败条目可展开查看失败文件详情。
7. **假设** 用户想要暂停某任务的自动调度，**当** 点击"禁用"开关，**则** 任务标记为禁用状态，下次调度时间不再触发。

### 用户故事 4 — 转码任务管理与监控（优先级：P2）

作为一名系统管理员，我希望在 Web 界面中创建独立转码任务、查看转码进度、重试上传失败的文件、手动清理残留临时文件，以便管理视频转音频的完整流程。

**为什么是此优先级**：转码是系统的增值功能，依赖于 P1 的存储引擎和同步任务基础。用户可以先完成存储和同步管理，再使用转码功能。但其独立测试价值完整。

**独立测试**：可以通过创建独立转码任务、观察进度变化、对上传失败的任务执行重试、触发残留文件清理来完整验证。

**验收场景**：

1. **假设** 用户已配置存储引擎并进入转码任务页面，**当** 点击"创建转码任务"，**则** 显示表单包含：源文件所在引擎（下拉选择）、目标引擎（下拉选择）、源文件路径、目标路径、目标格式（MP3/MP4/FLV）、码率（kbps，默认 128）。
2. **假设** 转码任务正在执行，**当** 用户在任务列表中查看，**则** 每条任务显示当前状态（等待中/转码中/上传中/完成/失败）、进度百分比、已耗时、预估剩余时间。
3. **假设** 某转码任务上传失败，**当** 用户点击"重试上传"按钮，**则** 系统尝试重新上传临时文件，成功或失败后更新任务状态。
4. **假设** 用户点击"清理残留文件"按钮，**当** 系统执行清理，**则** 页面显示已清理的临时文件数量，并刷新任务列表。
5. **假设** 用户进入转码任务详情，**当** 查看任务信息，**则** 显示源文件格式（通过魔数检测的结果）、输出文件大小、完整的处理时间线。

### 用户故事 5 — Webhook 规则配置与事件查看（优先级：P2）

作为一名使用录播姬的用户，我希望在 Web 界面中配置 Webhook 处理规则、查看接收到的 Webhook 事件记录，以便管理录播自动处理流程，无需直接编辑数据库。

**为什么是此优先级**：Webhook 集成为特定用户群体（录播姬用户）服务。它独立于同步和转码核心流程，可以作为独立功能切片存在。

**独立测试**：可以通过创建 Webhook 处理规则、查看接收到的 Webhook 事件列表、启用/禁用规则来完整验证。

**验收场景**：

1. **假设** 用户进入 Webhook 规则管理页面，**当** 点击"创建规则"，**则** 显示表单包含：规则名称、触发事件类型（FileClosed/SessionEnded/全部）、房间号过滤（可选，留空表示所有房间）、后续动作（同步至 AList/转 MP3/两者）、目标引擎和路径。
2. **假设** 用户已创建了规则，**当** 在规则列表中查看，**则** 每条规则显示名称、触发条件摘要、关联动作、启用状态。
3. **假设** 用户进入"Webhook 事件"页面，**当** 查看事件列表，**则** 以时间倒序显示接收到的所有 Webhook 事件，包含事件类型、发生时间、处理状态（待处理/处理中/已完成/失败）、关联的规则名称。
4. **假设** 用户想临时暂停某个处理规则，**当** 点击"禁用"开关，**则** 规则标记为禁用，后续匹配的 Webhook 事件不再触发处理动作。
5. **假设** 录播姬尚未发送过任何 Webhook 事件，**当** 用户查看事件列表，**则** 页面显示"暂未接收到任何 Webhook 事件"的空状态提示，并展示录播姬 Webhook 配置引导（URL 格式、配置步骤）。

### 用户故事 6 — 系统概览仪表板（优先级：P3）

作为一名系统管理员，我希望登录后首先看到一个系统概览仪表板，汇总关键指标（活跃任务数、今日处理文件数、成功率、存储空间概况），以便快速了解系统运行状态，无需逐页查看各模块。

**为什么是此优先级**：仪表板是锦上添花的功能，提升使用体验。用户可以直接从各功能页面获取所有信息，仪表板仅提供便捷的汇总视图。

**独立测试**：可以通过登录后查看仪表板显示的统计数据（活跃同步任务数、今日转码完成数、Webhook 事件数等）来验证，数据应与各模块详情页一致。

**验收场景**：

1. **假设** 用户已登录且系统中有运行中的任务，**当** 进入仪表板，**则** 显示：活跃同步任务数量、等待中的转码任务数量、今日处理文件总数、最近 24 小时任务成功率。
2. **假设** 用户点击仪表板中"活跃同步任务"卡片，**当** 跳转到同步任务管理页面，**则** 自动筛选显示正在运行的任务。
3. **假设** 当前没有活跃任务，**当** 用户进入仪表板，**则** 显示"系统当前空闲，所有任务已完成"的友好提示。

### 边界情况

- **浏览器兼容性**：页面必须在 Chrome、Firefox、Edge 的最近 3 个主要版本中正常显示和交互。
- **长时间运行任务**：转码或同步任务可能需要数小时，前端页面应定期轮询状态（默认 5 秒间隔），并支持手动刷新，不依赖 WebSocket。
- **网络中断**：当浏览器与后端服务器连接中断时，前端应显示"连接失败，正在重试"的提示，并在连接恢复后自动重试失败的请求。
- **空数据状态**：所有列表页面在没有数据时应显示友好的空状态提示和引导操作按钮（如"创建第一个同步任务"），而非空白页面。
- **大量数据渲染**：当同步任务包含超过 10,000 个文件时，执行详情页的文件列表应分页加载，每页不超过 100 条，避免浏览器卡顿。
- **表单输入校验**：所有表单应对必填字段、格式校验（如 cron 表达式、URL 格式）在前端提交前进行预校验，减少无效请求。
- **并发操作冲突**：当用户同时在多个浏览器标签页中打开管理界面时，对同一资源的修改应以后提交的为准，并在提交成功时显示提示。
- **页面刷新后状态保持**：用户在任务详情页刷新浏览器后，应能继续查看同一任务的信息（URL 路由应包含任务 ID）。
- **移动端访问**：管理界面主要为桌面浏览器设计（最小宽度 1024px），但应保证基本可读性在平板设备上（768px 及以上）。
- **会话过期**：用户登录会话过期后，前端应自动跳转到登录页面并显示"会话已过期，请重新登录"提示，保留用户之前浏览的页面路径以便登录后恢复。

## 需求 *（强制）*

### 功能需求

#### 认证与会话管理

- **FR-001**：Web 界面必须提供登录页面，用户输入用户名和密码后通过 HTTP Basic Authentication 调用后端 API 进行认证。
- **FR-002**：前端必须在所有 API 请求中携带 Basic Auth 凭据（用户名:密码的 Base64 编码），凭据存储在浏览器会话中（sessionStorage 或内存）。
- **FR-003**：Web 界面必须通过双重保障机制处理会话超时：(1) 前端监听用户鼠标/键盘交互事件（点击、移动、按键），30 分钟无交互后主动清除认证凭据并跳转到登录页面；(2) 后端在 30 分钟内未收到该用户 API 请求时返回 HTTP 401，前端据此作为服务端兜底触发重新登录。
- **FR-004**：前端必须在收到 HTTP 401 响应时自动跳转到登录页面，并在登录成功后恢复跳转前的页面路径。

#### 全局交互规范

- **FR-005**：Web 界面必须提供统一的导航结构——左侧固定侧边栏（240px）为主导航方式，包含系统概览、存储引擎、同步任务、转码任务、Webhook 规则、Webhook 事件的入口。平板设备（768px 宽度）下侧边栏折叠为顶部导航。
- **FR-006**：所有数据列表页面必须支持分页展示，默认每页 20 条记录。
- **FR-007**：所有创建/编辑表单必须在提交前进行前端校验（必填字段、格式校验），并在提交后显示操作结果反馈（成功/失败提示）。
- **FR-008**：所有删除操作必须先弹出确认对话框，用户确认后才执行删除，且对话框必须显示被删除对象的名称。
- **FR-009**：系统必须在每个 API 请求期间显示加载状态指示（按钮禁用 + 加载动画），请求完成后恢复正常状态。

#### 存储引擎管理页面

- **FR-010**：系统必须提供存储引擎列表页面，以表格形式展示所有引擎（名称、API 地址、用户名、创建时间、最近更新时间），凭据字段（密码/Token）脱敏显示。
- **FR-011**：系统必须提供存储引擎创建/编辑表单，字段包括：引擎名称（必填，最长 100 字符）、API 地址（必填，必须为有效 URL 格式）、用户名（必填）、密码或 Token（必填）。
- **FR-012**：表单必须对 API 地址进行 URL 格式校验（前端预校验），对空字段进行必填校验。
- **FR-013**：用户必须能够对任意引擎执行连接测试，测试结果以醒目颜色（绿色成功/红色失败）和文字反馈显示在列表中。

#### 同步任务管理页面

- **FR-014**：系统必须提供同步任务列表页面，以表格形式展示所有任务（名称、源引擎、目标引擎、同步模式、执行计划、启用状态、最后执行时间、最近执行状态）。
- **FR-015**：系统必须提供同步任务创建/编辑表单，字段包括：任务名称（必填）、源引擎（下拉选择已有引擎，必填）、目标引擎（下拉选择已有引擎，必填）、源路径（必填，以 `/` 开头）、目标路径（必填，以 `/` 开头）、同步模式（下拉选择：仅新增/全同步/移动，默认仅新增）、执行计划类型（单选：间隔执行/cron 表达式/手动触发）、间隔秒数（当选择间隔执行时必填）、cron 表达式（当选择 cron 时必填）、文件排除规则（可选，多行文本输入，每行一个模式）、冲突处理策略（下拉选择：覆盖/跳过/重命名，必填，默认值"跳过"）、"同步后转 MP3"开关（默认关闭）。
- **FR-016**：当用户输入 cron 表达式时，系统必须在输入框旁实时显示人类可读的说明和下一次执行时间预览。
- **FR-017**：用户必须能够对单个任务执行以下操作：手动触发执行、启用/禁用定时调度、查看执行历史。
- **FR-018**：任务执行时，系统必须展示实时进度信息：已完成文件数/总文件数、当前处理文件名、同步速度（文件/秒）、预估剩余时间。进度数据通过前端定期轮询更新（默认 5 秒间隔）。

#### 同步任务执行历史

- **FR-019**：系统必须在任务详情页提供"执行历史"标签，以分页表格展示：开始时间、结束时间、状态（运行中/成功/失败/部分成功/中断）、总文件数、成功数、失败数。
- **FR-020**：执行记录中失败数大于 0 的条目必须可展开，展开后显示失败文件列表（文件名、失败原因）。

#### 转码任务管理页面

- **FR-021**：系统必须提供转码任务列表页面，以表格展示所有转码任务（ID、源文件路径、目标路径、目标格式、进度百分比、状态、创建时间）。
- **FR-022**：系统必须提供创建独立转码任务的表单，字段包括：源文件所在引擎（下拉选择）、目标引擎（下拉选择）、源文件路径（必填）、目标路径（必填）、目标格式（下拉选择：MP3/MP4/FLV）、码率（kbps，默认 128）。
- **FR-023**：执行中的转码任务必须显示实时进度信息（进度百分比、当前状态：等待中/转码中/上传中、已耗时、预估剩余时间），通过前端定期轮询更新（默认 5 秒间隔）。
- **FR-024**：系统必须在上传失败的转码任务行中提供"重试上传"按钮。
- **FR-025**：系统必须在转码任务页面提供"清理残留临时文件"按钮，点击后触发后端清理，并显示已清理文件数量的反馈。

#### Webhook 规则管理页面

- **FR-026**：系统必须提供 Webhook 规则列表页面，以表格展示所有规则（规则名称、触发事件类型、房间号过滤、后续动作、启用状态）。
- **FR-027**：系统必须提供 Webhook 规则创建/编辑表单，字段包括：规则名称（必填）、触发事件类型（下拉多选或单选：FileClosed/SessionEnded/全部）、房间号过滤（可选，数字输入）、后续动作（多选：同步至 AList/转 MP3）、目标引擎（下拉选择，条件必填——当动作包含同步时）、目标路径（条件必填）。
- **FR-028**：用户必须能够启用/禁用单个规则。

#### Webhook 事件查看页面

- **FR-029**：系统必须提供 Webhook 事件列表页面，以时间倒序分页展示所有接收到的 Webhook 事件（事件类型、EventId、发生时间、关联规则、处理状态）。
- **FR-030**：当没有任何 Webhook 事件时，页面必须显示空状态提示和录播姬 Webhook 配置引导信息（包含 Webhook 接收端点 URL 格式说明）。

#### 系统概览仪表板

- **FR-031**：系统必须提供仪表板页面，显示以下统计卡片：活跃同步任务数（当前正在运行）、等待中转码任务数、今日处理文件总数、最近 24 小时任务成功率（百分比）。
- **FR-032**：仪表板的统计卡片必须可点击，点击后跳转到对应功能模块的列表页面（如点击活跃同步任务数跳转到同步任务列表）。
- **FR-033**：仪表板必须在没有活跃任务时显示友好提示"系统当前空闲"。

#### 全局技术需求

- **FR-034**：Web 界面必须与后端 API 由同一服务端口提供，无需部署额外的 Web 服务器或配置跨域访问。
- **FR-035**：前端页面必须通过调用现有 REST API 端点实现所有功能，不新增专用 API 端点（仪表板统计数据除外——后端可能需要提供聚合统计接口）。
- **FR-036**：所有用户界面文本必须使用简体中文（按钮、标签、提示信息、错误消息、空状态提示）。

### 关键实体

- **登录会话（LoginSession）**：代表用户登录状态，包含用户名、认证凭据（Base64 编码的用户名:密码）、最后活动时间戳。存储于浏览器会话中，30 分钟无操作后过期。
- **导航状态（NavigationState）**：代表用户当前浏览的页面路径和标签页状态，通过 URL 路由参数持久化，支持页面刷新后恢复。
- **表单校验规则（ValidationRule）**：定义各表单字段的校验逻辑（必填、格式、长度等），在前端提交前执行，减少无效 API 请求。

## 成功标准 *（强制）*

### 可衡量的结果

- **SC-001**：用户可以在首次使用时，不查看任何文档即可完成存储引擎配置和同步任务创建的完整流程，操作时间不超过 10 分钟。
- **SC-002**：所有页面在首次加载时（包含 20 条数据记录的列表页）完全渲染时间不超过 3 秒。
- **SC-003**：所有表单的提交操作（创建/编辑）在正常网络条件下 5 秒内完成并显示结果反馈（成功或失败提示）。
- **SC-004**：用户在 Web 界面中完成的管理操作（创建引擎、创建任务、触发执行等）100% 可通过后端 API 的响应验证其正确性，前端不改变业务数据。
- **SC-005**：Web 界面在 7 个主要功能页面（仪表板、存储引擎、同步任务、执行历史、转码任务、Webhook 规则、Webhook 事件）之间切换时，响应时间不超过 2 秒。
- **SC-006**：表单在前端提交前拦截 95% 以上的常见输入错误（空必填字段、非法 URL 格式、非法 cron 表达式），减少无效 API 请求。

## 假设

- **A1**：Web 前端经 Vite 构建后作为静态资源嵌入 Spring Boot 应用中（输出到 `src/main/resources/static/app/`），与后端运行在同一进程和同一端口，无需额外 Web 服务器或 CORS 配置。
- **A2**：前端采用 React 19 + TypeScript + Vite + Tailwind CSS 技术栈实现。与纯 HTML/CSS/JS 或 Vue.js CDN 方案相比，React + TypeScript 提供类型安全和更强的组件化能力，Vite 提供高效构建，Tailwind CSS 消除手写 CSS 维护负担。此选择已经过 plan 阶段完整评估（理由详见 research.md R1 和 plan.md 复杂性追踪）。
- **A3**：目标用户使用桌面浏览器（分辨率 1024×768 及以上），不支持移动端小屏触屏交互。平板设备（768px 宽度及以上）保持基本可用性。
- **A4**：前端认证采用 HTTP Basic Authentication 方式，凭据（用户名:密码 Base64 编码）通过 `Authorization` 请求头在每个 API 请求中携带。
- **A5**：前端页面路由使用 React Router v7 Hash 路由（`createHashRouter`），无需服务端 fallback 配置。
- **A6**：轮询间隔默认为 5 秒，用户不可配置此值（保持简单）。
- **A7**：仪表板统计数据通过后端新增的聚合统计 API 端点（`GET /api/dashboard/stats`）获取。
- **A8**：前端代码注释和用户界面文本均使用简体中文。
- **A9**：前端不直接操作数据库或文件系统，所有数据操作通过 REST API 完成。
- **A10**：本功能是对现有全部后端功能的 UI 层补充，不修改任何现有 API 契约（仪表板统计 API 为纯新增）。
