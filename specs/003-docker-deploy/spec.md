**功能分支**：`003-docker-deploy`

**创建日期**：2026-06-19

**状态**：草案

**输入**：用户描述："需要支持docker部署"

## 用户场景与测试 *（强制）*

### 用户故事 1 — 开发者通过 Docker 一键构建并运行应用（优先级：P1）

作为一名运维人员或开发者，我希望能够使用标准 Docker 命令构建应用镜像并启动容器，无需手动安装 Java 21、Maven 或配置环境变量，即可在任意支持 Docker 的机器上运行 AList-Media-Sync 服务。

**为什么是此优先级**：这是 Docker 部署的核心能力。没有可用的镜像构建和启动能力，其他所有 Docker 相关功能（数据持久化、多环境配置）都无法发挥作用。

**独立测试**：可以通过在未安装 Java/Maven 的机器上执行 `docker build` 构建镜像，然后执行 `docker run` 启动容器，访问应用健康检查端点返回正常状态来验证。交付价值：开发者无需配置 Java 开发环境即可部署应用。

**验收场景**：

1. **假设** 开发者拥有项目源代码且本地安装了 Docker，**当** 在项目根目录执行 `docker build -t alist-media-sync .`，**则** 镜像构建成功（多阶段构建：第一阶段编译打包应用，第二阶段仅包含运行时环境），最终镜像大小不超过 300MB。
2. **假设** 镜像已构建成功，**当** 执行 `docker run -d -p 8080:8080 alist-media-sync`，**则** 容器启动并在 30 秒内对外提供 HTTP 服务，访问 `http://localhost:8080/actuator/health` 返回健康状态。
3. **假设** 容器正在运行，**当** 执行 `docker stop <容器ID>` 和 `docker rm <容器ID>`，**则** 容器被正常停止并删除，H2 数据库中的数据不丢失（数据保存在命名卷中）。
4. **假设** 容器首次启动，**当** 应用日志输出，**则** 所有日志输出到标准输出（stdout），可通过 `docker logs` 查看。

---

### 用户故事 2 — 使用 Docker Compose 编排部署（优先级：P2）

作为一名运维人员，我希望通过 `docker-compose.yml` 文件一键启动应用及其依赖服务，无需记忆复杂的 `docker run` 参数，并且能够方便地管理配置和数据卷。

**为什么是此优先级**：Docker Compose 是 Docker 生态的标准编排工具，它简化了多容器应用的部署管理。对于本项目的单容器场景，Compose 仍提供了声明式配置的便利性。

**独立测试**：可以通过在任意装有 Docker Compose 的机器上执行 `docker compose up -d`，观察服务是否正常启动并可访问来验证。

**验收场景**：

1. **假设** 开发者拥有项目源代码且本地安装了 Docker Compose，**当** 在项目根目录执行 `docker compose up -d`，**则** 服务在后台启动，30 秒内可通过 `http://localhost:8080` 访问应用。
2. **假设** 服务已通过 Docker Compose 启动，**当** 修改 `docker-compose.yml` 中的端口映射配置，执行 `docker compose down && docker compose up -d`，**则** 服务以新配置重新启动。
3. **假设** 服务正在运行，**当** 执行 `docker compose logs -f`，**则** 可以实时查看应用日志。
4. **假设** 开发者希望更新应用版本，**当** 拉取新代码后执行 `docker compose build --no-cache && docker compose up -d`，**则** 使用新镜像重新启动服务，旧容器被替换。

---

### 用户故事 3 — 通过环境变量灵活配置应用（优先级：P3）

作为一名运维人员，我希望通过环境变量（而非修改配置文件）来配置应用的关键参数，例如数据库路径、AList API 地址、服务端口等，以适应不同部署环境（开发、测试、生产）的需求，无需重新构建镜像。

**为什么是此优先级**：环境变量配置是容器化部署的最佳实践（符合十二要素应用原则），但其价值在简单部署场景中不如基础镜像构建和 Compose 编排直接。适合作为增强项。

**独立测试**：可以通过启动容器时传入不同环境变量值，验证应用行为是否随环境变量变化来测试。

**验收场景**：

1. **假设** 默认配置下应用监听 8080 端口，**当** 启动容器时传入 `SERVER_PORT=9090` 环境变量，**则** 应用监听 9090 端口。
2. **假设** H2 数据库默认存储路径为 `./data`，**当** 启动容器时传入 `DATA_DIR=/opt/alist-media-sync/data` 环境变量，**则** 数据库文件保存在 `/opt/alist-media-sync/data` 路径下。
3. **假设** 日志级别默认为 INFO，**当** 启动容器时传入 `LOGGING_LEVEL=DEBUG` 环境变量，**则** 应用日志详细程度变为 DEBUG 级别。
4. **假设** 部署人员需要设置 AList 服务器地址，**当** 启动容器时传入 `ALIST_BASE_URL=https://alist.example.com` 环境变量，**则** 应用使用该地址作为默认的 AList 服务器连接地址。

### 边界情况

- 当 Docker 守护进程未运行或不可用时，构建和启动命令应给出清晰的错误提示（而非 Java 堆栈跟踪）。
- 当容器内存不足时（例如分配的内存小于应用所需最小值 256MB），应用应在日志中明确发出警告。
- 当容器接收到 SIGTERM 信号时（`docker stop`），应用应在 15 秒内优雅关闭（完成正在处理的同步任务，释放数据库连接）。
- 当数据卷挂载路径不存在或权限不足时，容器应在启动时给出明确错误信息，而非静默失败。
- 当多个容器实例同时运行并挂载同一数据卷时，内嵌数据库可能因文件锁冲突而无法启动——应在文档中明确说明此限制。
- 当镜像构建过程中依赖下载失败（网络问题）时，构建应失败并输出明确的错误信息，方便诊断。

## 需求 *（强制）*

### 功能需求

- **FR-001**：系统 MUST 提供 Dockerfile，采用多阶段构建（构建阶段编译打包 + 运行阶段仅包含运行时依赖），最终镜像仅包含运行时所需文件。
- **FR-002**：系统 MUST 提供 `docker-compose.yml` 文件，支持通过 `docker compose up` 一键启动服务。
- **FR-003**：Docker 镜像 MUST 基于轻量级 Java 21 运行时基础镜像，确保 Java 21 兼容性且镜像体积最小化。
- **FR-004**：容器 MUST 暴露 HTTP 服务端口（默认 8080），支持通过 `PORT` 环境变量自定义。
- **FR-005**：容器 MUST 支持通过环境变量配置应用运行参数（至少包括服务端口、数据库路径、日志级别），无需修改配置文件或重新构建镜像。
- **FR-006**：应用数据库文件 MUST 存储在可挂载的数据目录中（默认 `/app/data`），确保容器重启后数据不丢失。
- **FR-007**：容器 MUST 提供健康检查端点，使 Docker 和编排工具能够判断服务是否就绪。
- **FR-008**：`docker-compose.yml` MUST 定义命名卷用于持久化数据库数据，并在注释中说明数据备份方法。
- **FR-009**：`.dockerignore` 文件 MUST 排除不必要的文件（如 `.git`、`node_modules`、本地构建产物等），缩小构建上下文，加速构建。
- **FR-010**：容器内的应用 MUST 以非 root 用户运行（安全性要求），避免潜在的安全风险。
- **FR-011**：镜像构建和容器启动的完整步骤 MUST 记录在项目 README 或部署文档中。

### 关键实体

- **Docker 镜像**：包含完整可运行的 AList-Media-Sync 应用，对外暴露 HTTP 端口，通过环境变量接收配置。镜像分为两个构建阶段：构建阶段（包含构建工具和 Java 开发环境，产出可执行文件）和运行阶段（仅包含 Java 运行时，从构建阶段复制可执行文件）。
- **Docker 数据卷**：用于持久化 H2 数据库文件，独立于容器生命周期。在 Docker Compose 中定义为命名卷 `alist-media-sync-data`。
- **环境变量配置**：将应用配置项映射为容器环境变量（遵循外部化配置约定），覆盖默认配置文件中的值。

## 成功标准 *（强制）*

### 可衡量的结果

- **SC-001**：开发者可以在 5 分钟内完成从源码到容器运行的完整流程（`docker build` + `docker run`），无需安装 Java 或 Maven。
- **SC-002**：最终 Docker 镜像大小不超过 250MB，以控制存储和网络传输成本。
- **SC-003**：容器从执行 `docker run` 到健康检查端点返回就绪状态的时间不超过 30 秒（冷启动）。
- **SC-004**：容器优雅关闭时间不超过 15 秒（从接收 SIGTERM 到进程退出）。
- **SC-005**：通过环境变量修改端口或数据库路径后，应用启动成功率 100%（在 Docker 环境中）。
- **SC-006**：容器重启后，已持久化到数据卷的 H2 数据库数据完整无损（通过重启前后数据对比验证）。

## 假设

- 目标部署环境为 x86-64 架构的 Linux 服务器（或 macOS/Windows 上的 Docker Desktop），使用轻量级 Java 21 JRE 基础镜像。
- 当前 H2 数据库为内嵌模式，无需独立的数据库容器。如果未来切换到外部数据库（如 PostgreSQL），Docker Compose 文件可扩展。
- Docker Compose 编排仅针对单容器部署（本项目当前为单体应用），不涉及多容器集群编排（如 Kubernetes）。
- 应用默认配置中的外部服务地址（如 AList API 地址）由部署者在启动时通过环境变量提供，不在镜像中硬编码。
- 项目使用构建包装器（Wrapper）脚本进行构建，因此构建阶段无需在基础镜像中预装构建工具。
- `.dockerignore` 遵循项目构建惯例，排除版本控制目录、本地构建产物、IDE 配置文件等。
