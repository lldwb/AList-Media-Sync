# 功能规格：转码临时文件可配置后缀

**功能分支**：`002-transcode-temp-suffix-config`

**创建日期**：2026-06-19

**状态**：已澄清

**输入**：用户描述："转码过程中产生的临时文件需要使用特殊的后缀（如.lldwb 这种特殊标识后缀），这个后缀在配置文件里面可以配置，临时文件存储在服务器上，转码后上传到目标路径。"

## 澄清

### 会话 2026-06-19

- Q: 自动重试上传是否在本功能范围内？ → A: 自动重试属于后续功能，本功能仅实现手动重试支持（提供 API/接口让用户触发重试），自动重试推迟到独立功能规划
- Q: 残留临时文件的清理策略？ → A: 启动时自动扫描 + 手动清理：系统每次启动时扫描临时目录，对带有配置后缀的残留文件进行清理（删除），同时保留手动清理接口
- Q: 是否需要配置并发转码任务上限？ → A: 可配置的并发上限：在配置文件中新增并发上限配置项（默认值为 CPU 核心数），超出上限的转码任务排队等待
- Q: 临时文件是否需要设置文件系统权限？ → A: 最小权限原则：临时目录权限设为仅运行用户可读写（POSIX: 0700，Windows: 仅当前用户），目录内的临时文件权限设为仅运行用户可读写（POSIX: 0600）
- Q: 启动时清理是否应该区分"已完成转码待上传"和"转码中断"的文件？ → A: 启动时全部清理：系统启动时无条件清理临时目录下所有带配置后缀的文件，不论其关联任务状态。上传失败的文件需在系统重启后重新执行完整转码

## 用户场景与测试 *（强制）*

### 用户故事 1 — 配置自定义临时文件后缀（优先级：P1）

作为一名系统管理员，我希望能够在配置文件中自定义转码临时文件的后缀标识（如 `.lldwb`），以便临时文件在服务器文件系统中具有醒目、可识别的标记，方便运维排查和文件清理。

**为什么是此优先级**：这是本功能的核心——将硬编码的后缀变为可配置项。没有此能力，后续的流程改进（服务器暂存→上传）无法建立在可配置基础上。

**独立测试**：可以通过修改配置文件中的后缀设置、重启系统、触发一个转码任务，验证生成的中间临时文件是否使用了配置的后缀来测试。

**验收场景**：

1. **假设** 用户在配置文件中将临时文件后缀设置为 `.lldwb`，**当** 系统启动并加载配置，**则** 所有后续转码任务产生的临时文件使用 `.lldwb` 作为后缀。
2. **假设** 用户未在配置文件中指定临时文件后缀，**当** 系统启动，**则** 使用默认后缀 `.tmp`。
3. **假设** 用户设置了包含点号前缀的后缀如 `.transcoding`，**当** 系统生成临时文件，**则** 文件名格式为 `原文件名.ext.transcoding`。
4. **假设** 用户设置了不包含点号的后缀如 `lldwb`，**当** 系统生成临时文件，**则** 自动补充点号前缀，文件名格式为 `原文件名.ext.lldwb`。

---

### 用户故事 2 — 临时文件服务器本地暂存（优先级：P1）

作为一名关注转码可靠性的用户，我希望转码过程中的临时文件存储在服务器本地磁盘上，转码完成后再上传到目标 AList 存储，这样即使上传过程中发生网络故障，已转码完成的文件也不会丢失，可以重试上传。

**为什么是此优先级**：将临时文件暂存在服务器本地是"转码后上传到目标路径"的前提。这是架构层面的改进，直接提升转码可靠性。

**独立测试**：可以通过触发转码任务、在转码过程中检查服务器临时目录是否存在带配置后缀的中间文件、转码完成后验证文件被上传至目标 AList 路径，且服务器本地临时文件被清理来测试。

**验收场景**：

1. **假设** 用户已配置临时文件目录路径，**当** 转码任务开始处理一个视频文件，**则** 转码输出先写入服务器本地临时目录，文件使用配置的后缀（如 `.lldwb`）。
2. **假设** 转码成功完成且临时文件已生成，**当** 系统将文件上传到目标 AList 存储，**则** 上传成功后删除服务器本地临时文件。
3. **假设** 转码成功完成但上传到目标 AList 存储失败（网络中断），**当** 上传失败，**则** 服务器本地临时文件在当前会话中保留，任务记录中标记上传失败，支持用户在当前会话中手动触发重试上传。若系统重启，启动时自动清理所有残留临时文件，用户需重新执行转码任务。
4. **假设** 转码过程中系统宕机，**当** 系统重启，**则** 系统启动时自动扫描临时目录，识别并清理带有配置后缀的残留临时文件，完成清理后记录日志（清理数量）；同时保留手动清理接口供管理员随时调用。

### 用户故事 3 — 临时文件后缀配置校验（优先级：P2）

作为一名系统管理员，我希望系统在启动时对临时文件后缀配置进行合法性校验，防止配置了非法后缀（如空字符串、包含路径分隔符等）导致文件系统异常。

**为什么是此优先级**：校验是防御性措施，避免配置错误导致运行时问题。虽然重要，但不影响核心功能流程。

**独立测试**：可以通过在配置文件中填入各种非法后缀值（空字符串、`/`、`\`、`..`、仅空格等），验证系统启动时是否给出明确的配置错误提示来测试。

**验收场景**：

1. **假设** 用户在配置文件中设置了空字符串作为后缀，**当** 系统启动，**则** 系统回退使用默认后缀 `.tmp` 并记录警告日志。
2. **假设** 用户设置了包含路径分隔符的后缀（如 `/tmp` 或 `\temp`），**当** 系统启动，**则** 系统拒绝启动并给出明确错误提示："临时文件后缀包含非法字符：[具体字符]"。
3. **假设** 用户设置了仅包含点号的后缀（如 `.`），**当** 系统启动，**则** 系统回退使用默认后缀 `.tmp` 并记录警告日志。

### 边界情况

- 临时文件目录磁盘空间不足时，转码任务如何处理？系统应在转码开始前检查临时目录可用空间（至少保留预估输出文件大小的 1.5 倍空间），不足时拒绝启动转码并告警。
- 配置的后缀字符串过长（如超过 50 个字符）时如何处理？系统应将后缀长度限制在 50 个字符以内，超出时截断或使用默认值并记录警告。
- 临时文件目录不存在或无写入权限时如何处理？系统应在启动时检查并尝试创建目录，若失败则拒绝启动并记录错误。
- 多个转码任务同时写入同一个临时文件路径时如何处理？临时文件命名应包含唯一标识（如 UUID 或任务 ID）避免冲突。
- 临时文件上传完成后，删除操作失败（如文件被锁定）时如何处理？系统应记录警告日志并在后续重复执行删除操作（每次系统启动时扫描、或通过手动清理接口触发）。

## 需求 *（强制）*

### 功能需求

#### 临时文件后缀配置

- **FR-001**：系统必须支持通过配置文件设置转码临时文件的后缀标识。
- **FR-002**：系统必须在未配置后缀时使用默认后缀 `.tmp`。
- **FR-003**：系统必须自动处理后缀的点号前缀——若用户配置的值不以点号开头，系统自动补充点号；若配置的值以点号开头，直接使用。
- **FR-004**：系统必须在启动时对临时文件后缀配置进行合法性校验：后缀不得包含路径分隔符（`/`、`\`）、不得为仅点号（`.`）、不得包含空字符；校验失败时系统应拒绝启动并输出明确的错误信息。

#### 临时文件本地暂存与上传

- **FR-005**：系统必须将转码输出的临时文件写入服务器本地磁盘的临时目录（路径通过配置项指定，默认为系统临时目录下的程序专属子目录）。写入的临时文件必须设置权限为仅运行用户可读写（POSIX 系统设为 0600，Windows 仅当前用户）。
- **FR-006**：系统必须在临时文件命名中包含唯一标识，确保并发转码任务的临时文件不会相互冲突。
- **FR-007**：系统必须支持通过配置文件设置最大并发转码任务数（`max-concurrent-transcode`），默认值为服务器的 CPU 核心数；当待执行转码任务数超过上限时，超出任务排队等待。
- **FR-008**：系统必须在转码成功后将临时文件上传到目标存储的目标路径。
- **FR-009**：系统必须在上传成功后删除服务器本地的临时文件。若删除失败（文件被锁定等情况），记录警告日志并在后续启动扫描或手动清理时重试删除。
- **FR-010**：系统必须在上传失败时保留服务器本地临时文件，在任务执行记录中标记上传失败状态及失败原因，提供接口支持用户在当前会话中手动触发重试上传。若系统重启，启动时自动清理所有残留临时文件，用户需重新执行转码任务。自动重试机制不在本功能范围内，推迟到后续独立功能规划。
- **FR-011**：系统必须在转码开始前检查临时目录可用磁盘空间（至少需大于预估输出文件大小的 1.5 倍），空间不足时拒绝启动转码并在任务结果中记录明确错误信息。
- **FR-012**：系统必须在启动时检查临时文件目录是否存在，若不存在则尝试创建；创建时设置目录权限为仅运行用户可读写（POSIX 系统设为 0700，Windows 仅当前用户）；若创建失败（无权限等），系统应拒绝启动并输出错误信息。
- **FR-013**：系统必须在每次启动时扫描临时文件目录，无条件清理所有带有配置后缀的残留临时文件（不论其关联任务状态），记录清理数量到日志；同时提供手动清理接口，支持管理员随时触发残留文件扫描与清理。
- **FR-014**：系统必须将临时文件名中的临时后缀去掉后作为最终文件名在上传时使用。即：最终上传到目标存储的文件名不包含临时后缀标识。

### 关键实体

- **转码配置（TranscodeConfig）**：应用程序配置实体，包含临时文件后缀（`temp-suffix`，默认 `.tmp`）、临时文件目录路径（`temp-dir`，默认系统临时目录子目录）、后缀最大长度限制（50 字符）、最大并发转码任务数（`max-concurrent-transcode`，默认值为 CPU 核心数）。
- **临时文件（TempFile）**：代表转码过程中产生的临时输出文件，包含本地路径、唯一标识、关联的转码任务引用、创建时间、文件大小、状态（转码中/转码完成/上传中/上传完成/失败）。

## 成功标准 *（强制）*

### 可衡量的结果

- **SC-001**：用户可以在配置文件中的一行配置中更改转码临时文件后缀，无需修改任何代码。
- **SC-002**：转码产生的临时文件在上传到目标 AList 存储之前 100% 保留在服务器本地磁盘上。
- **SC-003**：网络中断导致上传失败时，在当前系统会话中已转码完成的临时文件不丢失，用户可在网络恢复后手动重试上传（系统重启后临时文件被清理，需重新转码）。
- **SC-004**：并发执行转码任务时（最多不超过配置的并发上限），各任务的临时文件不会因命名冲突而相互覆盖。
- **SC-005**：用户更改临时文件后缀设置后，系统在 1 分钟内即可完成重启并使新后缀生效。

## 假设

- **A1**：临时文件暂存于运行本系统的服务器本地磁盘上，该服务器具有足够的磁盘空间用于临时存储转码输出文件。
- **A2**：目标路径指 AList 存储中的路径，上传操作通过 AList API 实现，与现有同步功能的上传机制一致。
- **A3**：临时文件目录的默认路径位于系统临时目录（如 Linux 的 `/tmp/alist-media-sync/transcode`、Windows 的 `%TEMP%\alist-media-sync\transcode`）下，用户可通过配置项覆盖。
- **A4**：本功能是对现有同步/转码流程的增强，不改变同步任务的基本逻辑，仅修改转码输出文件的暂存位置和上传时机。
- **A5**：临时文件命名中的唯一标识应全局唯一，确保在多任务并发场景下不会发生冲突。
- **A6**：本功能遵循项目章程中定义的分层架构、数据完整性、配置外部化等核心原则。
