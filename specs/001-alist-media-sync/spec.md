# 功能规格：AList 媒体同步与转码工具

**功能分支**：`001-alist-media-sync`

**创建日期**：2026-06-19

**状态**：草案

**输入**：用户描述："我想做一个alist的同步工具参考（https://github.com/dr34m-cn/taosync），扩展功能：
1、同步并且同步完成进行视频转mp3的功能（可选择）
2、接收mikufans录播姬(https://github.com/BililiveRecorder/BililiveRecorder )录播完成状态并且进行转换的功能

## 用户场景与测试 *（强制）*

### 用户故事 1 — 多存储间文件同步（优先级：P1）

作为一名拥有多个 AList/OpenList 存储的用户，我希望能够创建同步任务，将文件从一个存储自动同步到另一个存储，以便在多个网盘之间保持一致的文件副本。

**为什么是此优先级**：这是系统的核心能力，所有其他功能（转码、录播集成）都依赖于此基础。没有同步能力，系统无法提供任何价值。

**独立测试**：可以通过配置两个 AList 存储、创建一个同步任务并观察文件从源存储复制到目标存储来完整测试。交付价值：用户无需手动下载再上传即可在多存储间同步文件。

**验收场景**：

1. **假设** 用户已配置了两个 AList 存储引擎（源和目标），**当** 用户创建一个"仅新增"模式的同步任务并执行，**则** 目标存储中不存在于源存储的文件被复制到目标存储，已存在的文件被跳过。
2. **假设** 用户已配置了两个 AList 存储引擎，**当** 用户创建一个"全同步"模式的同步任务并执行，**则** 目标存储的文件与源存储完全一致（新增缺失文件，删除多余文件）。
3. **假设** 用户创建了一个定时同步任务（cron 表达式），**当** 到达指定时间，**则** 系统自动执行同步任务。
4. **假设** 同步任务正在执行中，**当** 用户查看任务状态，**则** 可以看到当前同步进度（已完成文件数/总文件数、同步速度、预估剩余时间）。
5. **假设** 同步过程中某个文件传输失败，**当** 任务完成，**则** 系统记录失败文件的详细信息（文件名、失败原因、时间戳）并在任务结果中展示。

### 用户故事 2 — 同步后自动视频转 MP3（优先级：P2）

作为一名经常同步视频文件的用户，我希望在同步完成后能够自动将视频文件转换为 MP3 音频格式，以便在移动设备上节省空间收听内容，而不需要手动使用其他转码工具。

**为什么是此优先级**：此功能为用户提供增值体验，但依赖于 P1 同步功能。即使没有此功能，用户仍可从同步中受益。

**独立测试**：可以通过创建一个同步任务并启用"同步后转 MP3"选项，放入测试视频文件，执行同步后检查目标存储中是否生成了对应的 MP3 文件来测试。

**验收场景**：

1. **假设** 用户在创建同步任务时启用了"同步后转 MP3"选项，**当** 同步任务完成并将视频文件复制到目标存储后，**则** 系统自动将视频文件转换为 MP3 格式（128kbps 码率），并将生成的 MP3 文件存放到目标存储的指定目录。
2. **假设** 用户在创建同步任务时未启用"同步后转 MP3"选项，**当** 同步任务完成，**则** 仅完成文件同步，不执行任何转码操作。
3. **假设** 同步任务启用了转 MP3 选项且同步了 100 个视频文件，**当** 转码过程中第 50 个文件转码失败，**则** 系统继续处理剩余文件，并在任务结果中记录失败文件的详细信息。
4. **假设** 目标存储中已存在与待生成 MP3 同名的文件，**当** 转码完成，**则** 系统根据用户配置的策略处理冲突（覆盖、跳过、重命名）。
5. **假设** 用户在 AList 存储中已有视频文件，**当** 用户创建独立转码任务（指定源视频文件和目标格式），**则** 系统直接对指定文件进行转码，无需关联同步任务。

### 用户故事 3 — 接收录播姬 Webhook 并自动处理（优先级：P2）

作为一名使用 mikufans录播姬 录制 B 站直播的用户，我希望录播姬在录制完成时自动通知本系统，由系统自动将录制的视频文件同步到我的 AList 存储并转换为 MP3，实现从录制到存档的全自动化。

**为什么是此优先级**：此功能为特定用户群体（B站直播录制用户）提供高度自动化的体验。它与 P2 转码功能同级，可独立于 P1 同步功能存在。

**独立测试**：可以通过启动录播姬、配置其 Webhook URL 指向本系统、录播姬完成一次录制后观察本系统是否自动接收通知、拉取录制文件、同步至 AList 并进行转码来测试。

**验收场景**：

1. **假设** 用户在系统中配置了录播姬 Webhook 接收端点并关联了一个处理规则，**当** 录播姬发送 `FileClosed`（文件关闭）事件到本系统，**则** 系统记录该事件并在录制会话结束后开始处理录制文件。
2. **假设** 系统收到录播姬 `SessionEnded`（录制结束）事件，**当** 处理规则配置为"同步至 AList 并转 MP3"，**则** 系统自动将本次录制的所有视频文件同步到指定的 AList 存储，然后转换为 MP3。
3. **假设** 系统收到录播姬 Webhook 请求，**当** 请求内容符合 Webhook v2 协议但缺少必要字段，**则** 系统返回 HTTP 4xx 错误并记录异常日志。
4. **假设** 录播姬因网络问题重发了同一个 Webhook 事件（EventId 相同），**当** 系统检测到重复事件，**则** 跳过重复处理，返回 HTTP 200 表示已处理。
5. **假设** 用户配置了多个录播姬实例的处理规则，**当** 不同录播姬同时发送 Webhook，**则** 系统并行处理各实例的事件，互不干扰。

### 用户故事 4 — 实时转码状态监控（优先级：P3）

作为一名关注处理进度的用户，我希望能够实时查看当前正在进行的转码任务状态，包括进度百分比、当前处理的文件、预计完成时间等，以便合理安排后续操作。

**为什么是此优先级**：此功能提升用户体验，但不影响核心业务流程。用户可以通过其他方式（如日志）获取状态信息。

**独立测试**：可以通过触发一个包含大量视频文件的转码任务，在任务执行期间查询任务状态 API 来验证实时状态信息的准确性。

**验收场景**：

1. **假设** 有一个转码任务正在执行，**当** 用户查询任务状态，**则** 系统返回当前处理的文件名、已完成数量、总数量、预估剩余时间。
2. **假设** 转码任务已完成，**当** 用户查询任务状态，**则** 系统返回完成状态、总处理数量、成功数量、失败数量及失败详情。

### 边界情况

- 当源存储文件大小超过目标存储可用空间时会发生什么？系统应在同步前进行空间预估检查，若空间不足则提前终止并告警。
- 系统如何处理网络中断导致的大文件传输失败？应支持断点续传或自动重试机制。
- 当录播姬 Webhook 发送频率过高（如同时多个直播间结束录制）时，系统如何防止资源耗尽？应采用任务队列限流处理。
- 视频文件格式不支持转 MP3（如文件已损坏）时如何处理？系统应记录失败日志并跳过，不影响其他文件的转码。
- 当 AList 存储 API 不可用（离线或限流）时，同步任务如何处理？应支持重试策略（指数退避），超时后标记任务失败。
- 转码过程中系统宕机重启后，未完成的转码任务如何恢复？系统应持久化任务状态，重启后恢复未完成的任务。
- 当多个同步任务（或同步任务与 Webhook 触发的处理）同时向同一目标存储的相同路径写入文件时，系统应检测目标路径冲突并拒绝执行后续任务（返回明确错误提示），由用户调整任务计划避免冲突。架构上保留策略扩展空间（终止/跳过/覆盖）。
- 当用户手动触发一个正在执行中的同步任务时会发生什么？系统应检测到任务实例仍在运行，跳过本次触发，记录日志并向用户提示"任务正在执行中，请稍后再试"。
- 系统重启后正在执行的同步/转码任务如何处理？系统应在启动时自动将"运行中"状态的任务标记为"中断"，并从数据库重新注册所有已启用的定时任务（cron/间隔），确保定时调度不丢失。

## 澄清

### 会话 2026-06-19

- Q: 当两个同步任务同时向同一目标路径写入时的冲突处理策略？ → A: 默认检测并拒绝执行后续任务，返回明确错误提示；架构上保留终止/跳过/覆盖的策略扩展空间供后期优化。
- Q: Web 管理界面的认证机制？ → A: 用户名+密码认证，配置文件设置凭据，所有管理 API 和 Web 界面需登录后访问。
- Q: 同一任务手动触发时，若上一次实例仍在执行？ → A: 跳过本次触发，记录日志并向用户提示任务正在执行中。
- Q: H2 数据库运行模式（内存 vs 文件持久化）？ → A: 文件持久化模式（jdbc:h2:file），重启后保留所有配置、任务历史和中间状态。
- Q: 转码是否支持独立运行（不依赖同步任务）？ → A: 是，转码以三种形态存在——独立转码任务、同步任务的可选后置步骤、Webhook 触发处理动作。
- Q: 定时任务调度在系统重启后如何恢复？ → A: 数据库持久化调度——cron/间隔配置持久化到数据库，系统启动时自动从数据库重新注册所有已启用的定时任务。

## 需求 *（强制）*

### 功能需求

#### 存储引擎管理

- **FR-001**：系统必须允许用户添加、编辑、删除、测试连接 AList/OpenList 存储引擎（每个引擎包含名称、URL、认证凭据）。
- **FR-002**：系统必须安全存储引擎的认证凭据（不可逆加密），敏感信息不得以明文形式存储或传输。

#### 同步任务管理

- **FR-003**：系统必须允许用户创建同步任务，每个任务包含：名称、源引擎、目标引擎、源路径、目标路径、同步模式（仅新增/全同步/移动）、执行计划（间隔执行/cron 表达式/手动触发）。
- **FR-004**：系统必须支持任务的启用、禁用、手动执行、编辑、删除操作。手动触发时，若同一任务的上一次实例仍在执行中（状态为"运行中"），系统必须跳过本次触发，记录日志，并向用户返回明确提示（"任务正在执行中，请稍后再试"）。
- **FR-005**：系统必须在同步任务执行过程中实时记录进度（已完成文件数、总文件数、当前文件、同步速度、预估时间）。
- **FR-006**：系统必须支持文件排除规则（按文件名模式或目录过滤不同步的文件）。
- **FR-007**：系统必须记录每次任务执行的完整历史（开始时间、结束时间、状态、处理文件数、成功数、失败数、失败文件详情）。

#### 视频转码

- **FR-008**：系统必须支持三种转码触发方式：(1) 同步任务的后置步骤（可开启/关闭），(2) Webhook 处理规则的关联动作，(3) 独立的转码任务（用户直接选择已有视频文件进行转码）。转码目标格式可选择 MP3、MP4、FLV。
- **FR-009**：系统必须使用 JAVE2（Java Audio Video Encoder）库封装 FFmpeg 调用，而非直接通过命令行进程调用 FFmpeg。JAVE2 原生二进制文件通过 Maven 依赖按平台自动选择（Windows/Linux）。
- **FR-010**：系统必须在转码输出时使用临时文件后缀策略（如 `.tmp`），转码成功后将临时文件重命名为最终扩展名，防止转码中断产生损坏文件被误认为完成。
- **FR-011**：当转码作为同步任务后置步骤触发时，系统必须仅在同步成功完成后才启动转码流程（同步 → 转码顺序执行）。当转码作为独立任务触发时，系统直接对指定视频文件进行转码。
- **FR-012**：系统必须支持文件冲突处理策略（覆盖、跳过、重命名），在转码输出文件与目标位置已有文件冲突时按策略处理。
- **FR-013**：系统必须通过文件内容（魔数/Magic Bytes）检测输入视频格式，而非依赖文件扩展名。支持的输入视频格式至少包括 FLV、MP4、M4V。

#### 文件树遍历与批量处理

- **FR-014**：系统必须支持递归深度优先遍历源目录树，收集所有符合条件的视频文件，形成待处理文件列表。
- **FR-015**：系统必须在遍历阶段检查目标文件是否已存在（相同路径+输出格式扩展名），若已存在则跳过（需考虑冲突策略配置），避免不必要的重复转码。
- **FR-016**：系统必须采用两阶段处理模式：第一阶段单线程递归扫描收集文件列表，第二阶段多线程并行处理文件列表中的文件。
- **FR-017**：系统必须通过可配置的线程池实现多线程并行转码，线程池参数包括：核心线程数（默认 8）、最大线程数（默认 32）、队列容量（默认 64）、拒绝策略（CallerRunsPolicy——队列满时回退到调用者线程执行，提供自然背压）。
- **FR-018**：系统必须通过 Spring `@Async` 注解标记异步转换方法，使用 `@Lazy @Autowired` 自注入代理模式确保 AOP 代理生效，每个文件转换返回一个 `CompletableFuture`。

#### 录播姬 Webhook 集成

- **FR-019**：系统必须提供一个 HTTP 端点接收录播姬 Webhook v2 格式的 POST 请求（包括 `SessionStarted`、`FileOpening`、`FileClosed`、`SessionEnded`、`StreamStarted`、`StreamEnded` 事件类型）。
- **FR-020**：系统必须根据 `EventId` 字段去重，确保同一事件不会重复处理。
- **FR-021**：系统必须在收到 Webhook 请求后立即返回 HTTP 2xx 状态码（处理逻辑异步执行），以满足录播姬的超时要求。
- **FR-022**：系统必须允许用户创建 Webhook 处理规则，规则包含：规则名称、事件触发条件（按房间号过滤可选）、后续动作（同步至 AList/转换为 MP3/两者）。
- **FR-023**：系统必须支持将录播姬录制的文件同步至指定的 AList 存储目标，并在同步完成后（若规则配置）自动执行视频转码。

#### 通用需求

- **FR-024**：系统必须提供任务执行历史的查询接口，支持按时间范围、任务类型（同步/转码/Webhook 处理）、状态过滤。
- **FR-025**：系统必须记录所有关键操作日志（同步任务启动/完成、文件传输失败、转码开始/完成/失败、Webhook 接收/处理失败）。
- **FR-026**：系统必须支持配置任务执行记录和日志的保留天数，过期记录自动清理以控制存储占用。
- **FR-027**：系统必须支持通过配置控制转码任务的并发数量，防止系统资源耗尽。
- **FR-028**：系统必须为所有管理 API 和 Web 界面提供用户名+密码认证保护，凭据通过配置文件设置，未经认证的请求必须返回 HTTP 401 并拒绝访问。录播姬 Webhook 接收端点作为外部回调接口，不要求认证。

### 关键实体

- **存储引擎（StorageEngine）**：代表一个 AList/OpenList 实例，包含名称、API 地址、用户名、加密后的密码/Token、连接状态、创建/更新时间。
- **同步任务（SyncTask）**：代表一个同步作业，包含名称、源引擎引用、目标引擎引用、源路径、目标路径、同步模式、转 MP3 开关、冲突策略、排除规则、执行计划（cron/间隔/手动）、启用状态、最后执行时间。
- **任务执行记录（TaskExecution）**：代表一次任务执行的完整记录，包含关联任务引用、任务类型（同步/转码/Webhook处理）、开始/结束时间、状态（运行中/成功/失败/部分成功）、总文件数、成功数、失败数、失败详情列表。
- **Webhook 处理规则（WebhookRule）**：代表一个录播姬 Webhook 的处理配置，包含规则名称、触发事件类型（`FileClosed`/`SessionEnded`）、房间号过滤条件（可选，为空表示所有房间）、后续动作（同步/转码/两者）、目标引擎引用、目标路径。
- **Webhook 事件记录（WebhookEvent）**：代表一条接收到的 Webhook 事件，包含 EventId（去重键）、事件类型、时间戳、原始数据、处理状态（待处理/处理中/已完成/失败）。
- **转码任务（TranscodeTask）**：代表一个视频转 MP3 的转码作业，包含源文件路径、目标文件路径、视频格式、目标码率、状态、进度、关联的同步任务或 Webhook 规则引用。

## 成功标准 *（强制）*

### 可衡量的结果

- **SC-001**：用户可以在 5 分钟内完成首个存储引擎的配置和一个基础同步任务的创建。
- **SC-002**：同步任务的文件传输成功率在正常网络环境下达到 99% 以上（排除外部服务不可用导致的失败）。
- **SC-003**：单个视频文件（1GB 以内）的 MP3 转码在 5 分钟内完成（标准硬件配置下）。
- **SC-004**：系统在收到录播姬 Webhook 请求后 1 秒内返回 HTTP 响应（异步处理不计入响应时间）。
- **SC-005**：系统支持同时处理至少 5 个转码任务而不显著降级（转码速度下降不超过 20%）。
- **SC-006**：用户可以在任务执行历史中准确定位并查看任何一次失败任务的详细失败原因，无需查阅原始日志文件。
- **SC-007**：系统在单次超过 10,000 个文件的同步任务中不发生崩溃或内存泄漏。

## 假设

- **A1**：用户拥有可用的 AList/OpenList 实例（v3+），本系统不集成 AList，与 TaoSync 一致需要用户自行部署 AList。
- **A2**：视频转码功能使用 JAVE2（`ws.schild:jave-core:3.5.0`）库封装 FFmpeg 调用，而非通过命令行进程调用。JAVE2 原生二进制文件（`jave-nativebin-win64`、`jave-nativebin-linux64`）作为 Maven 依赖按 `os.arch` 自动选择对应平台的 FFmpeg 二进制文件。
- **A3**：录播姬 Webhook 集成基于录播姬 Webhook v2 协议。系统仅处理来自录播姬 2.0.0+ 版本的请求。
- **A4**：目标用户为有一定技术能力的用户（能够部署 AList、录播姬等工具），系统提供 REST API 和管理 Web 界面。
- **A5**：v1 版本的录播姬集成聚焦于 `FileClosed` 和 `SessionEnded` 两种事件类型，其他事件类型（`StreamStarted`、`StreamEnded` 等）的记录和处理将在后续版本中扩展。
- **A6**：系统部署在用户可控的服务器或 NAS 上，不与公网直接交互。所有外部通信（AList API、录播姬 Webhook）在用户内网环境中进行。
- **A7**：文件同步操作通过调用 AList API 接口实现（文件列表获取、文件上传/下载、文件删除），不直接操作 AList 底层存储。同步操作的源和目标都是 AList 存储。本地文件系统不作为同步的源或目标。但录播姬录制文件存储在本地磁盘，转码操作直接操作本地文件系统（从录播姬录制目录读取 → 转码生成 MP3 → 通过 AList API 上传到目标存储）。
- **A8**：转码采用与 videoConversionMonitor 相同的两阶段模式：第一阶段单线程递归扫描收集文件列表（按魔数检测格式，检查目标是否已存在），第二阶段多线程并行转码（线程池：核心 8、最大 32、队列 64、CallerRunsPolicy）。
- **A9**：转码输出使用临时文件后缀（`.tmp`），成功后重命名为最终扩展名，防止转码中断产生损坏文件。
- **A10**：Spring `@Async` 通过 `@Lazy @Autowired` 自注入代理模式确保 AOP 生效，避免内部 `this.method()` 调用绕过代理。
- **A11**：H2 数据库使用文件持久化模式（`jdbc:h2:file`），数据持久化到磁盘文件。系统重启后任务配置、执行历史、中间状态全部保留，崩溃恢复场景可基于持久化状态重建。数据库文件存放于用户可配置的数据目录中。定时任务调度（cron/间隔）的配置持久化在数据库中，系统启动时自动从数据库重新注册所有已启用的定时任务，确保重启后定时调度不丢失。
