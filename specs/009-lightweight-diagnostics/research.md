# 阶段 0 研究：轻量诊断系统

## 决策 1：HTTP traceId 暴露位置

**决策**：所有后端 HTTP 响应统一携带 `X-Trace-Id` 响应头；请求如携带合法 `X-Trace-Id` 则沿用，否则后端生成新 traceId。错误响应仍保持现有 `ApiResult<T>` 契约，不强制把 traceId 放入响应体。

**理由**：响应头是非破坏性扩展，不影响现有前后端 JSON 契约，也适合浏览器、CLI、日志和 AI 排障引用。用户提出“响应头里（或者其他更好的地方）”，响应头是当前项目最小改动且可观测性好的位置。

**考虑的替代方案**：
- 放入 `ApiResult<T>`：会改变全局响应 DTO，影响前端类型和现有契约，返工范围更大。
- 仅写入日志：用户或前端无法从请求结果直接获取 traceId，不利于反馈问题。
- 使用 W3C `traceparent`：更标准但对当前单实例轻量排障过重；可在未来兼容，但本功能先使用可读的 `X-Trace-Id`。

## 决策 2：traceId 生命周期与日志注入

**决策**：请求级 traceId 由过滤器管理，进入请求时写入 MDC，请求结束时清理；后台任务由任务入口生成或继承 traceId，并在任务执行线程中显式设置和清理 MDC。

**理由**：SLF4J MDC 是当前日志栈的内置能力，能让 Logback pattern 自动输出 traceId，不需要修改每一条日志调用。后台任务与 HTTP 请求不同线程执行，必须显式传递上下文，避免并发污染。

**考虑的替代方案**：
- 手动给每条日志增加 traceId 参数：侵入性大，容易遗漏。
- 引入分布式追踪库：不符合轻量化和 YAGNI。

## 决策 3：错误日志分流

**决策**：通过 Logback 配置单独的 ERROR appender，将 ERROR 及以上日志写入 `logs/error.log`，普通应用日志继续写入原日志目标。

**理由**：错误分流属于日志配置能力，避免在业务代码中重复写文件。`logs/error.log` 可被诊断包直接采集，满足 AI 快速定位失败证据的需求。

**考虑的替代方案**：
- 业务代码捕获异常后手动写 error 文件：重复且容易和日志输出不一致。
- 仅从普通日志中过滤 ERROR：诊断生成时成本更高，也不符合用户明确要求。

## 决策 4：一键诊断入口覆盖形态

**决策**：本地开发、Docker、一体化启动包都提供一键诊断入口。入口可以是脚本，也可以调用受认证保护的后端诊断端点；计划优先脚本入口，必要时复用后端只读诊断服务。

**理由**：用户已澄清三种部署形态都必须覆盖。脚本能在应用异常不可访问时仍收集文件和环境信息；后端服务能复用脱敏与摘要生成逻辑。

**考虑的替代方案**：
- 仅提供 HTTP 端点：应用无法启动时不可用。
- 仅提供脚本：难以复用应用内配置解析和最近任务上下文。

## 决策 5：诊断包内容与脱敏策略

**决策**：诊断包包含 `summary.md`、错误日志摘录、普通日志位置、脱敏配置摘要、运行环境摘要和最近任务上下文。脱敏采用保守策略：字段名命中 password、token、secret、key、authorization、cookie、credential 等语义，或值疑似凭据时统一替换为 `***REDACTED***`。

**理由**：诊断包需要可直接交给 AI 或开发者，默认安全比保留完整值更重要；保留字段名和来源可继续定位配置项。

**考虑的替代方案**：
- 仅脱敏固定字段：容易遗漏新增配置或请求头。
- 完全移除敏感字段：排查人员无法判断配置是否存在或为空。

## 决策 6：诊断包刷新语义

**决策**：`diagnostics/latest` 始终表示最近一次诊断结果。生成新诊断包时先写入临时目录，完成后再替换 latest；失败时生成最小摘要说明失败原因。

**理由**：避免半成品诊断包误导用户；latest 语义简单，满足轻量化定位需求。

**考虑的替代方案**：
- 按时间戳长期归档：更完整但增加清理策略和存储管理复杂度，本功能不需要。
- 原地覆盖文件：中途失败会留下不一致结果。
