## 场景 1：源目录转码 UX 优化

### 1.1 文案修正

**验证步骤**：
1. 启动应用：`./mvnw spring-boot:run`
2. 打开浏览器访问 `http://localhost:8080/app/`
3. 登录（admin / admin123）
4. 导航至转码任务页面 → 点击"创建转码任务"
5. 查看复选框标签

**预期结果**：复选框标签显示为"源目录转码（输出至源文件所在目录）"，而非"原目录转码"

### 1.2 表单字段条件隐藏

**验证步骤**：
1. 在转码任务创建表单中，勾选"源目录转码"复选框
2. 观察表单变化
3. 取消勾选

**预期结果**：
- 勾选后："目标存储引擎"下拉框和"目标文件路径"输入框立即隐藏（DOM 不可见）
- 取消勾选后：两个字段重新显示，恢复为必填状态

### 1.3 源目录转码路径计算

**验证步骤**：
1. 创建一个源目录转码任务：源引擎选择 AList，源路径 `/videos/sub/recording.flv`，目标格式 MP3
2. 触发转码执行
3. 查看输出文件路径

**预期结果**：输出文件路径为 `/videos/sub/recording.mp3`（源文件所在目录 / 源文件名（不含扩展名）.目标扩展名）

## 场景 2：转码任务批量操作

### 2.1 清理失败任务

**验证步骤**：
1. 确保存在至少 1 个失败状态的转码任务
2. 在转码任务列表页点击"清理失败任务"按钮
3. 确认对话框后观察

**预期结果**：
- 弹出确认对话框："确定要清理所有失败任务吗？此操作不可撤销。"
- 确认后按钮显示"清理中..."并禁用
- 完成后提示"已清理 N 个失败任务"
- 列表自动刷新

### 2.2 清理成功任务

**验证步骤**：
1. 确保存在至少 1 个 COMPLETED 状态的转码任务
2. 点击"清理成功任务"按钮

**预期结果**：
- 弹出确认对话框
- 确认后所有 COMPLETED 任务被删除
- 提示"已清理 N 个成功任务"

### 2.3 重试所有失败文件

**验证步骤**：
1. 确保存在至少 1 个可重试的失败任务（DOWNLOAD_FAILED/TRANSCODE_FAILED/UPLOAD_FAILED）
2. 点击"重试所有失败文件"按钮

**预期结果**：
- 弹出确认对话框："确定要重试所有失败文件吗？"
- 确认后按钮显示"重试中..."并禁用
- API 返回 202，提示"已提交 N 个任务进行重试，结果将通过实时更新推送"
- 通过 WebSocket 实时接收每个任务的重试结果

### 2.4 空列表边界情况

**验证步骤**：
1. 确保没有任何失败任务
2. 点击"清理失败任务"或"重试所有失败文件"

**预期结果**：显示提示"没有可操作的失败任务"

## 场景 4：WebSocket 实时推送

### 4.1 连接建立

**验证步骤**：
1. 打开浏览器开发者工具 → Network → WS 标签
2. 访问任意管理页面（如转码任务列表）
3. 观察 WebSocket 连接

**预期结果**：
- 建立到 `ws://localhost:8080/ws/events` 的连接
- 连接状态为 101 Switching Protocols
- 页面不再发送周期性 HTTP 轮询请求（30 秒内 HTTP 请求数减少 80%+）

### 4.2 实时进度推送

**验证步骤**：
1. 打开转码任务列表页面
2. 在另一个标签页触发转码任务执行
3. 观察第一个标签页的列表变化

**预期结果**：任务状态和进度实时更新，无需手动刷新页面

### 4.3 断线重连

**验证步骤**：
1. 打开转码任务列表页面，确认 WebSocket 已连接
2. 重启后端服务（模拟断线）
3. 观察前端行为

**预期结果**：
- 页面显示"连接中断，正在重连..."提示
- 5 秒内自动重连成功（指数退避：1s→2s→4s→...→30s）
- 重连后恢复实时更新

### 4.4 认证失败处理

**验证步骤**：
1. 修改密码使认证凭据失效
2. 刷新页面

**预期结果**：
- WebSocket 连接失败（401）
- 引导用户重新登录
- 不退化为 HTTP 轮询

### 4.5 连接数上限

**验证步骤**：
1. 配置 `app.websocket.max-connections=2`
2. 打开 3 个浏览器标签页访问管理页面

**预期结果**：第 3 个标签页的 WebSocket 连接被拒绝（HTTP 429）

## 场景 5：自动重试

### 5.1 瞬时故障自动重试

**验证步骤**：
1. 配置 `app.retry.max-auto-retries=3`
2. 模拟网络中断（如临时关闭 AList 服务）
3. 触发同步任务执行
4. 恢复网络
5. 观察任务状态

**预期结果**：
- 系统自动重试失败的文件（指数退避：1s→2s→4s）
- 重试成功后任务继续执行
- 日志显示重试次数和间隔

### 5.2 重试用尽标记最终失败

**验证步骤**：
1. 配置 `app.retry.max-auto-retries=2`
2. 模拟持续网络中断
3. 触发同步任务执行

**预期结果**：
- 重试 2 次后标记为最终失败
- 前端显示"重试 2/2"
- 用户可手动重试（手动重试不计入自动重试次数）

### 5.3 业务错误不重试

**验证步骤**：
1. 创建转码任务，源文件格式不支持
2. 触发转码

**预期结果**：
- 格式不支持异常不实现 `RetryableException`
- 直接标记为最终失败，不进行自动重试
- 日志显示"业务错误，不进行自动重试"
