**创建日期**：2026-06-20

**状态**：已排期

**输入**：用户描述："存储引擎配置移除用户名、策略模式支持本地文件系统、树状目录浏览组件、Webhook 路径显示优化、转码流程三步优化、Cron 图形化配置、文案统一替换"

## 澄清

### 会话 2026-06-20

- Q: 转码任务三步流程的状态模型设计？ → A: 8状态模型（待处理/下载中/下载失败/转码中/转码失败/上传中/上传失败/完成），失败为独立状态
- Q: 转码临时文件的生命周期管理策略？ → A: 上传成功后删除临时文件；失败状态下保留供重试；定时任务清理超过24小时的孤立临时文件
- Q: 树状目录浏览组件的加载方式与内容范围？ → A: 实时加载（非懒加载），只显示目录不显示文件，通过存储引擎接口的获取目录方法获取子目录
- Q: 存储引擎类型切换的UI约束方式？ → A: 编辑模式下类型字段禁用（灰色不可点击），其他字段可正常编辑
- Q: Webhook 规则中录播存储引擎的关联方式？ → A: 用户显式选择录播存储引擎（下拉选择器），不提供自动检测选项

## 用户场景与测试 *（强制）*

### 用户故事 1 — 存储引擎策略模式与本地路径支持（优先级：P1）

作为一个系统管理员，我希望能够配置多种类型的存储引擎（AList 远程存储和本地文件系统路径），并通过策略模式实现灵活切换，这样我可以根据实际部署环境选择最合适的存储方式，而不需要修改业务逻辑代码。

**为什么是此优先级**：策略模式是本次重构的基础架构变更，所有后续功能（转码流程优化、路径浏览等）都依赖存储引擎的抽象化。没有此基础，其他功能无法正确实现。

**独立测试**：可以通过创建不同类型的存储引擎（AList 类型、本地路径类型），分别执行同步和转码操作，验证两种引擎均可独立工作来完整测试。

**验收场景**：

1. **假设** 系统中无任何存储引擎，**当** 用户创建一个 AList 类型的存储引擎（填写地址和 Token，不填写用户名），**则** 存储引擎创建成功，连接测试通过
2. **假设** 系统中无任何存储引擎，**当** 用户创建一个本地路径类型的存储引擎（填写本地目录路径），**则** 存储引擎创建成功，连接测试通过（验证目录可读可写）
3. **假设** 系统中已有一个 AList 存储引擎和一个本地路径存储引擎，**当** 用户创建同步任务分别指定不同引擎作为源和目标，**则** 两个同步任务均可正常执行
4. **假设** 系统中已有存储引擎，**当** 用户编辑存储引擎配置，**则** 不再显示用户名字段，原有用户名数据被忽略，类型字段禁用（灰色不可点击），其他字段可正常编辑

### 用户故事 2 — 转码流程三步优化与任务过滤（优先级：P1）

作为一个系统管理员，我希望转码处理流程采用"下载 → 转码 → 上传"三步模式，并且在转码任务列表中只看到文件条目而非目录，这样转码过程更加清晰可靠，任务列表也更加整洁。

**为什么是此优先级**：转码是核心业务功能，当前流程直接在远程文件流上操作，三步模式可显著提升可靠性和可恢复性。任务列表过滤是直接影响用户体验的问题。

**独立测试**：可以通过创建转码任务并观察其执行流程（下载到临时目录、转码、上传到目标引擎）以及列表展示结果来完整测试。

**验收场景**：

1. **假设** 源存储引擎中有一个 FLV 视频文件，**当** 用户创建转码任务将其转为 MP3，**则** 系统先将源文件下载到临时目录，再在临时目录中执行转码，最后将转码结果上传到目标存储引擎，任务状态依次经过：待处理 → 下载中 → 转码中 → 上传中 → 完成（8状态模型：待处理/下载中/下载失败/转码中/转码失败/上传中/上传失败/完成）
2. **假设** 源存储引擎中有目录和文件混合的路径，**当** 系统扫描并列出转码候选条目，**则** 转码任务列表中仅显示文件类型条目，不显示目录
3. **假设** 转码过程中下载步骤失败，**当** 用户触发重试，**则** 任务状态从"下载失败"回到"下载中"，重新执行下载步骤
4. **假设** 转码过程中转码步骤失败，**当** 用户触发重试，**则** 任务状态从"转码失败"回到"转码中"，从临时目录中已下载的源文件重新执行转码，无需重新下载
5. **假设** 转码过程中上传步骤失败，**当** 用户触发重试，**则** 任务状态从"上传失败"回到"上传中"，从临时目录中的已转码文件重新上传，无需重新下载和转码
6. **假设** 转码任务的码率字段未填写，**当** 系统执行转码，**则** 使用系统默认码率进行转码
7. **假设** 转码任务完成（上传成功），**当** 系统清理临时文件，**则** 临时目录中的源文件和转码输出文件被删除
8. **假设** 临时目录中存在超过24小时的孤立临时文件，**当** 定时清理任务执行，**则** 孤立文件被自动删除

### 用户故事 3 — Webhook 规则路径显示优化与地址复制（优先级：P2）

作为一个系统管理员，我希望在 Webhook 规则的"转 MP3"配置项中清晰看到源端和目标端的存储引擎与路径组合，并能一键复制 Webhook 地址用于录播姬配置，这样我可以更直观地理解数据流向，并快速完成录播姬的 Webhook 对接。

**为什么是此优先级**：Webhook 是自动化触发的核心入口，路径显示优化直接影响配置正确性，地址复制减少手动拼写出错。但此功能依赖存储引擎策略模式完成后才能完整实现。

**独立测试**：可以通过创建和编辑 Webhook 规则，观察路径字段显示格式，以及点击复制按钮验证剪贴板内容来完整测试。

**验收场景**：

1. **假设** 用户正在编辑 Webhook 规则且选择了"转 MP3"或"同步 + 转 MP3"动作，**当** 查看源路径配置，**则** 显示为"录播存储引擎（下拉选择器） → 录播文件路径"的组合形式，用户必须显式选择录播存储引擎
2. **假设** 用户正在编辑 Webhook 规则且选择了"转 MP3"或"同步 + 转 MP3"动作，**当** 查看目标路径配置，**则** 显示为"目标存储引擎 → 目标文件路径"的组合形式，不再显示独立的目标目录路径字段
3. **假设** 用户在 Webhook 规则列表页面，**当** 点击"复制 Webhook 地址"按钮，**则** 系统将完整的录播姬 Webhook V2 URL 复制到剪贴板，格式为 `{服务地址}/api/webhook/v2`
4. **假设** 用户在 Webhook 规则列表页面，**当** 点击"复制 Webhook 地址"按钮后，**则** 页面显示复制成功的提示信息

### 用户故事 4 — 文件路径树状浏览组件（优先级：P2）

作为一个系统管理员，我希望在配置存储引擎路径时能通过树状目录组件浏览和选择文件路径，这样我不需要手动输入完整路径，降低配置出错的可能性。

**为什么是此优先级**：树状浏览组件显著改善路径配置体验，但需要后端提供目录树查询接口作为前置条件。属于体验优化，不影响核心功能可用性。

**独立测试**：可以通过在存储引擎配置或同步任务配置中打开路径选择器，浏览目录结构并选择路径来完整测试。

**验收场景**：

1. **假设** 用户配置存储引擎或同步任务需要填写路径，**当** 点击路径输入框旁的浏览按钮，**则** 弹出树状目录浏览组件
2. **假设** 树状目录组件已打开，**当** 用户点击某个目录节点，**则** 实时加载该目录的子目录列表（仅显示目录，不显示文件），通过存储引擎接口的获取目录方法获取
3. **假设** 树状目录组件已打开且当前为 AList 类型引擎，**当** 用户浏览目录，**则** 通过后端接口调用 AList 存储引擎的获取目录方法读取目录结构
4. **假设** 树状目录组件已打开且当前为本地路径类型引擎，**当** 用户浏览目录，**则** 通过后端接口调用本地路径存储引擎的获取目录方法读取本地文件系统的目录结构
5. **假设** 用户在树状目录中选中一个路径，**当** 点击确认，**则** 路径自动填入对应的输入框

### 用户故事 5 — Cron 表达式图形化配置（优先级：P3）

作为一个系统管理员，我希望在配置定时任务时能够通过图形化的下拉选择器来设置 Cron 表达式，而不是手动输入复杂的表达式语法，这样我无需了解 Cron 语法即可完成调度配置。

**为什么是此优先级**：图形化配置降低使用门槛，属于体验优化，不阻塞核心功能。Cron 手动输入方式仍然可用，图形化配置是补充而非替代。

**独立测试**：可以通过创建或编辑同步任务，使用图形化选择器配置 Cron 表达式，验证生成的表达式与预期一致来完整测试。

**验收场景**：

1. **假设** 用户正在创建或编辑同步任务且选择了 Cron 调度方式，**当** 查看 Cron 配置区域，**则** 显示分钟、小时、日、月、周五个独立的下拉选择器
2. **假设** 用户通过五个下拉选择器选择了具体值，**当** 完成选择，**则** 系统自动生成对应的 Cron 表达式并显示预览
3. **假设** 用户已通过图形化配置生成了 Cron 表达式，**当** 查看表达式预览，**则** 显示中文可读描述和下次执行时间

### 用户故事 6 — 文案统一替换（优先级：P3）

作为一个系统管理员，我希望在所有界面中看到统一的"存储引擎"术语，而不是"AList"混用的情况，这样我可以清晰地理解系统支持多种存储引擎类型，而不仅仅是 AList。

**为什么是此优先级**：文案统一是品牌和概念一致性的重要保障，配合策略模式重构，确保用户认知与系统能力一致。但属于展示层变更，不影响功能。

**独立测试**：可以通过遍历所有前端页面，确认不再出现"AList"相关文案（产品名称除外）来完整测试。

**验收场景**：

1. **假设** 系统前端所有页面正常加载，**当** 用户遍历存储引擎管理、Webhook 规则等页面，**则** 所有涉及存储引擎概念的文案均显示为"存储引擎"而非"AList"
2. **假设** 系统前端登录页面和侧边栏，**当** 用户查看产品名称，**则** 产品名称（如"AList Media Sync"）保持不变，不作为存储引擎概念替换

### 边界情况

- 当本地路径存储引擎配置的目录不存在时，系统返回错误提示，不自动创建目录（避免误操作创建不期望的目录结构）
- 当本地路径存储引擎配置的目录无读写权限时，连接测试返回明确的权限不足错误信息
- 存储引擎类型一旦创建后不可更改，编辑模式下类型字段禁用，如需更改类型需删除后重新创建
- 转码三步流程中下载步骤失败时，任务进入"下载失败"状态，不保留部分下载文件；用户可重试重新下载
- 转码三步流程中转码步骤失败时，任务进入"转码失败"状态，保留已下载的源文件以便重试转码
- 转码三步流程中上传步骤失败时，任务进入"上传失败"状态，保留已下载的源文件和转码输出文件以便重试上传
- 转码任务完成（上传成功）后，临时文件立即清理；超过24小时的孤立临时文件由定时任务自动清理
- 当 Cron 图形化配置的值无法生成有效 Cron 表达式时（如选择了不兼容的字段组合），系统禁用确认按钮并在配置区域下方显示红色提示文字"无法生成有效的 Cron 表达式，请调整配置"
- 当 Webhook 地址中服务地址未配置时，复制按钮使用当前页面的 origin 地址作为服务地址生成完整 URL；若页面 origin 为 localhost，则在复制时额外提示"当前使用本地地址，生产环境请配置服务地址"

## 需求 *（强制）*

### 功能需求

**存储引擎策略模式与本地路径支持：**

- **FR-001**：系统必须从存储引擎配置中移除用户名字段，存储引擎的连接与认证不再依赖用户名
- **FR-002**：系统必须使用策略模式实现存储引擎的动态切换，定义统一的存储引擎操作接口（文件列表、下载、上传、目录浏览等）
- **FR-003**：系统必须支持本地文件系统路径作为新的存储引擎类型，可执行文件列表、下载（本地复制）、上传（本地写入）和目录浏览操作
- **FR-004**：系统必须确保现有 AList 存储引擎的所有功能在策略模式重构后不受影响
- **FR-005**：存储引擎必须包含类型标识字段，用于区分 AList 类型与本地路径类型，类型一旦创建后不可更改（编辑模式下类型字段禁用）

**转码流程优化：**

- **FR-006**：系统必须在转码任务列表展示时过滤目录项，仅显示文件类型的条目
- **FR-007**：系统必须将文件转码处理流程优化为三步：先将源文件下载到临时目录 → 在临时目录中执行转码 → 将转码后的文件上传到目标存储引擎。转码任务状态采用8状态模型：待处理、下载中、下载失败、转码中、转码失败、上传中、上传失败、完成（含 3 对失败/重试状态 + 待处理 + 完成）。失败状态下保留临时文件供重试，上传成功后立即清理临时文件，另设定时任务清理超过24小时的孤立临时文件
- **FR-008**：系统必须优化 JAVE2 编解码参数配置：AudioAttributes/VideoAttributes 的 codec 设为 null 让 FFmpeg 自动选择编解码器（避免硬编码编解码器导致的兼容性问题）；bitrate 为 null 时使用系统默认码率（128 kbps）
- **FR-009**：创建转码任务时，目标格式为必填项，码率（kbps）为选填项，不填时使用系统默认码率

**Webhook 规则路径显示优化：**

- **FR-010**：Webhook 规则的"转 MP3"配置项中，源路径必须显示为"录播存储引擎（下拉选择器，用户显式选择） → 录播文件路径"的组合形式
- **FR-011**：Webhook 规则的"转 MP3"配置项中，目标路径必须显示为"目标存储引擎 → 目标文件路径"的组合形式，不再显示独立的目标目录路径字段
- **FR-012**：Webhook 规则页面必须提供"复制 Webhook 地址"按钮，可复制用于录播姬 Webhook V2 的完整 URL（格式：`{服务地址}/api/webhook/v2`）

**文件路径树状浏览：**

- **FR-013**：后端必须提供目录树查询接口，支持读取指定存储引擎（AList 或本地路径）的目录结构
- **FR-014**：前端必须实现文件路径选择的树状目录组件，采用实时加载方式（点击目录节点时通过存储引擎接口的获取目录方法获取子目录），仅显示目录不显示文件
- **FR-015**：树状目录组件必须支持选择路径后自动填入对应的输入框

**Cron 图形化配置：**

- **FR-016**：所有涉及 Cron 表达式配置的地方，必须增加可视化的图形化配置方式，使用分钟、小时、日、月、周五个独立下拉选择器组合
- **FR-017**：图形化配置生成的 Cron 表达式必须与现有 Cron 预览功能（中文描述 + 下次执行时间）联动

**文案统一替换：**

- **FR-018**：所有涉及"AList"的显示文案（存储引擎概念相关）必须统一替换为"存储引擎"，产品名称中的"AList"保持不变

### 关键实体

- **存储引擎（StorageEngine）**：代表一个存储后端实例。关键属性包括：名称、类型（AList/本地路径）、连接信息（AList 为地址+Token，本地路径为目录路径）、状态。移除用户名字段，新增类型字段。
- **Webhook 规则（WebhookRule）**：代表录播姬事件触发的自动化规则。路径相关字段从单一"目标目录路径"变更为"录播存储引擎 + 录播文件路径"和"目标存储引擎 + 目标文件路径"的组合。
- **转码任务（TranscodeTask）**：代表一个转码作业。码率字段从必填变为选填，新增系统默认码率配置。处理流程从两阶段变为三步（下载→转码→上传），采用8状态模型（待处理/下载中/下载失败/转码中/转码失败/上传中/上传失败/完成）。临时文件在上传成功后立即清理，失败状态下保留供重试，超过24小时的孤立临时文件由定时任务清理。

## 成功标准 *（强制）*

### 可衡量的结果

- **SC-001**：用户可以在 1 分钟内完成一个本地路径类型存储引擎的创建和连接测试（在本地开发环境下，无网络延迟时）
- **SC-002**：AList 类型和本地路径类型的存储引擎在同步和转码功能上的行为完全一致，用户无需关心底层差异
- **SC-003**：转码任务列表中不再出现目录条目，所有条目均为文件类型
- **SC-004**：转码流程中断后，用户可通过重试从失败步骤继续，无需重复已完成的步骤
- **SC-005**：90% 的用户无需查阅 Cron 语法文档即可通过图形化选择器完成调度配置（发布后用户调研指标，非构建性工作；构建性前置条件为：图形化选择器覆盖所有标准 Cron 场景且预览描述清晰可读）
- **SC-006**：用户可在 5 秒内通过"复制 Webhook 地址"按钮获取正确的录播姬 Webhook V2 URL
- **SC-007**：所有前端页面中不再出现以存储引擎概念出现的"AList"文案

## 假设

- 假设本地路径存储引擎的目录在配置时必须已存在，系统不自动创建目录（避免误操作创建不期望的目录结构）
- 假设本地路径存储引擎的连接测试验证目录的读写权限，权限不足时返回明确的错误信息
- 假设存储引擎类型一旦创建后不可更改，编辑模式下类型字段禁用（避免类型切换导致路径不兼容问题），如需更改类型需删除后重新创建
- 假设转码三步流程中下载失败时不保留部分下载文件，任务进入"下载失败"状态；转码失败时保留已下载的源文件，任务进入"转码失败"状态；上传失败时保留源文件和转码输出文件，任务进入"上传失败"状态
- 假设转码任务完成（上传成功）后临时文件立即清理，超过24小时的孤立临时文件由定时任务自动清理
- 假设系统默认码率为 128 kbps（与当前硬编码值一致），可通过配置文件调整
- 假设 Webhook 地址中的服务地址从应用配置中获取，如未配置则使用当前页面的 origin 地址
- 假设产品名称（"AList Media Sync"、"AList Sync"）中的"AList"不属于文案替换范围，保持原样
- 假设 Webhook 规则中录播存储引擎由用户显式选择（下拉选择器），不提供自动检测选项
