# 功能规格：一体化启动包

**功能分支**：`005-standalone-bootstrap`

**创建日期**：2026-06-20

**状态**：草案

**输入**：用户描述："请为 AList-Media-Sync 项目设计并生成一体化启动的完整实现方案（Docker 之外的方案）。将项目所有依赖整合为一个自包含的启动包，提供统一入口脚本，封装常见启动错误检测与修复，复用现有 Maven Wrapper 机制，在控制台输出可访问地址列表，并实现配置文件明文密码的首次启动自动加密。"

## 用户场景与测试 *（强制）*

### 用户故事 1 — 开箱即用的首次启动（优先级：P1）

作为一名刚下载 AList-Media-Sync 一体化启动包的用户，我希望解压后仅执行一条命令（或双击启动脚本）即可完成服务的完整启动，无需手动安装 Java 运行时、配置 Maven、安装 Node.js 或做任何其他环境准备工作，以便在 5 分钟内让系统跑起来。

**为什么是此优先级**：这是整个功能的核心价值——消除部署门槛。如果用户仍需手动安装依赖，一体化启动包就失去了意义。所有其他用户故事都依赖于此基础。

**独立测试**：可在一台全新安装的操作系统上（仅包含操作系统默认组件，未安装 Java/Maven/Node.js）解压启动包、执行启动脚本，验证服务是否成功启动并能通过浏览器访问管理界面。

**验收场景**：

1. **假设** 用户已下载一体化启动包并解压到本地目录，操作系统未预装 Java 运行时，**当** 用户执行 `start.bat`（Windows）或 `start.sh`（Linux），**则** 系统自动使用内置的 JRE 启动 Spring Boot 应用，前端资源从内置静态文件提供服务，无需联网下载任何额外依赖。
2. **假设** 启动包已解压，用户首次执行启动脚本，**当** 应用启动完成，**则** 控制台输出醒目的启动成功横幅，包含本地回环地址（`http://localhost:端口`、`http://127.0.0.1:端口`）和所有活跃网络接口地址，同时列出主要功能路径（管理界面 `/app/`、API `/api/`、健康检查 `/actuator/health`、H2 控制台 `/h2-console`）。
3. **假设** 用户修改了 `application.yaml` 中的应用端口为 `9090`，**当** 重新执行启动脚本，**则** 应用监听在 9090 端口，控制台输出的地址列表自动更新为正确端口。

### 用户故事 2 — 启动错误自动检测与友好提示（优先级：P1）

作为一名不熟悉 Java 生态的用户，我希望在启动过程中如果遇到常见问题（如端口被占用、配置文件损坏、数据目录无写入权限等），系统能自动检测并以中文给出明确的错误原因和解决建议，而不是看到一堆 Java 堆栈跟踪，以便我能自行排查问题而无需查阅文档或求助他人。

**为什么是此优先级**：启动错误的用户友好程度直接决定了非技术用户能否独立使用本系统。与 P1 核心故事同等重要，因为启动失败后的体验直接影响用户对产品可靠性的信任。

**独立测试**：可人为制造各种启动故障（占用端口、删除配置文件、设置数据目录为只读等），执行启动脚本，验证是否输出中文错误描述和修复建议。

**验收场景**：

1. **假设** 目标端口（如 8080）已被其他进程占用，**当** 用户执行启动脚本，**则** 系统检测到端口占用，输出中文提示："端口 8080 已被进程 [进程名] (PID: [编号]) 占用。建议：1) 修改 application.yaml 中的 server.port 配置项；2) 或终止占用进程后重试"，应用不启动。
2. **假设** `application.yaml` 配置文件被误删或损坏无法解析，**当** 用户执行启动脚本，**则** 系统检测到配置缺失，输出中文提示："配置文件 application.yaml 不存在或格式错误。建议：请检查 config/ 目录下的配置文件是否完整，可从模板文件 application.template.yaml 复制一份并修改。"
3. **假设** 数据目录（如 `./data`）所在的磁盘空间不足（剩余空间 < 100MB），**当** 用户执行启动脚本，**则** 系统检测到磁盘空间不足，输出中文警告："数据目录所在磁盘剩余空间不足 100MB，长期运行可能导致数据库写入失败。建议：清理磁盘空间或将数据目录迁移到空间充足的磁盘。是否继续启动？[y/N]"
4. **假设** 启动脚本检测到本地无内置 JRE 且系统也未安装 Java，**当** 用户执行启动脚本，**则** 系统输出中文提示："未检测到 Java 运行环境。本启动包已内置 JRE，但似乎在解压或复制过程中丢失。请重新下载完整的启动包。" 并退出。

### 用户故事 3 — 配置密码明文自动加密（优先级：P2）

作为一名运维人员，我希望在配置文件中直接填写明文密码（如 `password: "admin123"`），系统在首次启动时自动将其加密为 BCrypt 哈希值，后续启动无需我手动生成哈希，这样既能保证配置文件的人类可读性和易修改性，又能确保密码在运行时态的安全存储。

**为什么是此优先级**：提升了运维便利性和安全性，但不影响系统核心功能。即使没有此功能，管理员也可手动生成 BCrypt 哈希。但在自动部署场景中，明文配置能显著简化初始化流程。

**独立测试**：在 `application.yaml` 中设置 `app.auth.password: "test123"`（明文），启动系统，验证控制台日志输出"检测到明文密码，已自动加密"消息，再次启动系统，验证不再输出加密消息且用 `test123` 仍能正常登录管理界面。

**验收场景**：

1. **假设** 配置文件中 `app.auth.password` 设置为明文 `"admin123"`，**当** 应用首次启动，**则** 系统检测到该值非 BCrypt 格式（无 `{bcrypt}` 前缀），自动使用 BCrypt 算法加密密码并更新到配置文件中，控制台输出"检测到明文密码，已自动加密为 BCrypt 格式"。
2. **假设** 配置文件中 `app.auth.password` 已经是 `"{bcrypt}$2a$10$..."` 格式，**当** 应用再次启动，**则** 系统检测到密码已加密，跳过加密步骤，控制台不输出加密相关日志。
3. **假设** 用户修改了明文密码为新值 `"newpassword456"`，**当** 重新启动应用，**则** 系统重新加密新密码并更新配置文件，用户可使用新密码登录。
4. **假设** 配置文件中 `app.auth.password` 为空或未设置，**当** 应用启动，**则** 系统输出中文警告："认证密码未设置，管理后台将无法登录。请在 application.yaml 中配置 app.auth.password。"，但服务仍正常启动（API 端点的 401 保护依然生效）。

### 用户故事 4 — 一键打包生成启动包（优先级：P2）

作为开发者，我希望执行一条 Maven 命令或构建脚本即可自动完成整合打包（编译后端、构建前端、复制 JRE、生成启动脚本、打包为 zip/tar.gz），输出的启动包经过完整性校验后可直接分发给用户，无需我手动收集各个构建产物。

**为什么是此优先级**：这是开发者工作流的一部分。虽然可以手动打包，但自动化保证了每次构建的一致性，避免人为遗漏（如忘记复制某个依赖文件）。与 P2 同级的其他故事独立。

**独立测试**：从干净仓库执行打包命令，验证生成的 zip/tar.gz 文件包含完整的目录结构、启动脚本有可执行权限（Linux）、解压后启动脚本能正常运行。

**验收场景**：

1. **假设** 项目源码完整且通过所有测试，**当** 开发者执行 `./mvnw package -P bootstrap`（或专用打包脚本），**则** Maven 依次完成：前端构建（Vite）→ 后端编译打包 → 复制 JRE → 复制启动脚本 → 生成 zip/tar.gz 归档文件，控制台输出打包摘要（文件大小、包含的组件版本等）。
2. **假设** 打包过程完成，**当** 开发者将生成的 `alist-media-sync-standalone.zip` 解压到任意目录并执行启动脚本，**则** 服务正常启动，前端界面可访问。
3. **假设** 打包脚本检测到前端构建失败（如 TypeScript 编译错误），**当** 此情况发生，**则** 打包过程终止并输出错误详情，不生成不完整的启动包。

### 用户故事 5 — Docker 控制台地址输出（优先级：P3）

作为通过 Docker 部署的用户，我希望容器启动后能在 `docker logs` 中看到与服务网络可访问地址相同格式的输出，包括容器内部端口映射关系，以便与宿主机端口对应进行访问。

**为什么是此优先级**：该功能是对 Docker 部署体验的小幅增强。Docker 用户通常已经有一定技术能力，即使没有此功能也能通过 `docker ps` 和配置推断访问地址。但它提升了与一体化启动包体验的一致性。

**独立测试**：构建 Docker 镜像并启动容器，查看 `docker logs` 输出，验证是否包含与服务启动后相同格式的地址列表。

**验收场景**：

1. **假设** Docker 容器已通过 `docker compose up` 启动，**当** 应用启动完成，**则** `docker logs` 输出包含容器内监听端口和可访问路径列表，格式与一体化启动包一致。
2. **假设** Docker Compose 中配置了端口映射 `"9090:8080"`，**当** 查看启动日志，**则** 地址输出仍然显示容器内部端口（8080），并在端口旁注明"容器内部端口，宿主机映射端口请参阅 docker-compose.yml"。

### 边界情况

- 当用户的操作系统用户名包含空格或中文时，默认数据目录路径是否会出问题？启动脚本需确保所有路径使用引号包裹，配置文件中的路径需正确处理特殊字符。
- 当内置 JRE 版本与操作系统架构不匹配时（如 x86 JRE 在 ARM 机器上运行）会发生什么？启动脚本需检测架构并给出明确提示。
- 当用户同时运行多个启动包实例（不同端口）时会发生什么？每个实例应有独立的数据目录，通过环境变量或配置文件区分。启动脚本应在检测到已有实例运行时给出提示。
- 当配置文件被编辑器锁定（Windows 上常见）时，密码加密写回可能失败。系统需检测写入失败并给出明确中文提示，而非静默失败导致下次启动仍用明文。
- 当打包过程在中途被中断（如磁盘满）时，已有的临时文件如何清理？打包脚本应使用临时目录并在完成后清理，异常退出时也应尽量清理。
- 当启动脚本在符号链接或网络路径下执行时，路径推导可能出错。脚本需解析真实路径后再进行目录定位。

## 需求 *（强制）*

### 功能需求

#### 自包含启动包结构

- **FR-001**：启动包必须包含 Spring Boot 可执行 JAR（含全部后端依赖和内嵌 Web 服务器）、前端静态资源（Vite 构建产物已打入 JAR 或外部存放）、JAVA 运行时环境（JRE，与目标平台架构匹配）、配置文件（application.yaml 及模板文件）、数据库初始化脚本（如需要）、启动脚本（Windows `.bat` 和 Linux `.sh`）。
- **FR-002**：启动包解压后必须形成一个自包含的目录结构，所有文件位于该目录内，无需写入系统目录、注册表或用户目录即可运行（数据文件写入启动包内的 `data/` 子目录）。
- **FR-003**：启动包目录结构的顶层必须清晰可辨：启动脚本位于根目录，JRE 位于独立子目录（如 `runtime/`），配置文件位于 `config/` 子目录，可执行 JAR 位于 `lib/` 子目录。

#### 统一入口脚本

- **FR-004**：系统必须提供 `start.bat`（Windows）和 `start.sh`（Linux/Mac）两个启动脚本，用户双击或终端执行即可启动服务。脚本必须自动检测内置 JRE，无需用户设置 `JAVA_HOME` 环境变量。
- **FR-005**：启动脚本必须支持从环境变量读取配置覆盖（如 `SERVER_PORT`、`DATA_DIR`、`LOGGING_LEVEL` 等），与 Spring Boot 外部化配置机制兼容。脚本中也可通过设置变量传递给 Java 进程。
- **FR-006**：启动脚本必须能在任意工作目录下执行（即用户无需先 `cd` 到启动包目录），脚本通过自身所在位置定位启动包根目录并设置正确的相对路径（解析符号链接）。
- **FR-007**：启动脚本必须将 Java 进程的标准输出和标准错误输出打印到控制台，同时写入日志文件（`logs/app.log`），便于事后排查问题。

#### 启动前自检与错误处理

- **FR-008**：启动脚本在启动 Java 进程前必须执行预检查，包括：JRE 存在性检查、配置文件存在性检查、端口可用性检查、数据目录写入权限检查、磁盘剩余空间检查。每项检查失败时输出中文错误描述和修复建议。
- **FR-009**：端口占用检测必须识别占用进程的名称（Windows 通过 `netstat`，Linux 通过 `ss`/`lsof`）并在提示中包含进程名和 PID。如无法获取进程信息，仅输出端口号和通用停止建议。
- **FR-010**：磁盘剩余空间检查的阈值必须可配置（默认 100MB），低于阈值时给出警告并询问是否继续（交互模式），或直接退出（非交互模式/CI 环境）。
- **FR-011**：当检测到已有同应用的实例在运行时（通过端口检测或 PID 文件判断），启动脚本必须输出中文提示"检测到已有实例正在运行（PID: [编号]）"并询问是否强制重启。

#### 服务启动后地址输出

- **FR-012**：应用启动完成后，必须在控制台日志中输出醒目的启动成功横幅，包含以下信息：
  - 应用名称和版本号
  - 本地回环地址（`http://localhost:端口` 和 `http://127.0.0.1:端口`）
  - 所有活跃网络接口地址（如 `http://192.168.1.100:端口`）及其对应的网卡名称
  - 主要可访问路径列表：管理界面（`/app/`）、API 根路径（`/api/`）、健康检查端点（`/actuator/health`）、H2 控制台（`/h2-console`）
- **FR-013**：地址列表必须在应用端口监听就绪后自动输出——即在 Web 服务器完成端口绑定并能接收请求之后触发，而非在应用启动流程的早期阶段，确保输出的端口号是实际监听的端口。
- **FR-014**：Docker 容器内启动时也必须输出相同格式的地址列表。当检测到运行在容器环境时，在地址旁注明"容器内部端口"。

#### 配置文件密码加密

- **FR-015**：`application.yaml` 中的 `app.auth.password` 配置项必须支持明文值（如 `"admin123"`）。系统在启动时检测该值是否为 BCrypt 格式（通过 `{bcrypt}` 前缀或 BCrypt 哈希特征判断），若不是则自动使用 BCrypt 算法加密并回写到配置文件。
- **FR-016**：密码加密过程必须记录日志：加密成功时输出中文日志"检测到明文密码，已自动加密为 BCrypt 格式"；写入配置文件失败时输出中文错误日志"密码加密失败：无法写入配置文件 [具体原因]。请手动生成 BCrypt 哈希值替换明文密码"。
- **FR-017**：如果配置文件中 `app.auth.password` 为空或未设置，系统必须在日志中输出中文警告，提醒用户设置密码，但不应阻止应用启动（认证拦截器仍然生效，所有管理 API 将因无有效凭据而不可访问）。
- **FR-018**：密码加密功能必须与外部化配置兼容——如果密码通过环境变量（`APP_AUTH_PASSWORD`）覆盖，加密逻辑同样对覆盖后的值生效。

#### 打包构建

- **FR-019**：系统必须提供自动化打包机制，执行以下步骤：① 构建前端静态资源；② 编译后端并打包为可独立运行的 JAR 文件；③ 下载/复制目标平台的 JRE；④ 组合目录结构；⑤ 生成启动脚本（占位符替换为实际 JRE 路径和 JAR 文件名）；⑥ 打包为 `.zip`（Windows）和 `.tar.gz`（Linux）。
- **FR-020**：打包过程必须复用项目现有的 Maven Wrapper（`.mvn/wrapper/`）机制。前端构建步骤可通过 Maven 插件（如 `frontend-maven-plugin`）或 Shell 脚本在 Maven 生命周期中触发。打包配置不得破坏现有的 Docker 构建流程和开发环境构建流程。
- **FR-021**：打包产物（zip/tar.gz）必须在打包完成后由脚本进行完整性校验——解压产物到临时目录，验证关键文件存在（JAR、JRE 可执行文件、配置文件、启动脚本），校验通过后输出打包摘要。

#### 跨平台与分发

- **FR-022**：启动包必须区分 Windows（`.zip`）和 Linux（`.tar.gz`）两种分发格式。每种格式内置对应平台的 JRE。Linux 启动脚本中的换行符必须为 LF，防止因 CRLF 导致的脚本执行失败。
- **FR-023**：Linux 启动脚本在解压后必须具有可执行权限。如因传输丢失了可执行位，用户执行 `sh start.sh` 也应能正常运行。

### 关键实体

- **启动包**：一体化交付物的顶层概念，包含运行 AList-Media-Sync 所需的全部组件。包含 JRE 运行时、可执行 JAR、前端静态资源、配置文件、启动脚本、数据目录模板。以压缩归档形式分发（`.zip` 用于 Windows，`.tar.gz` 用于 Linux）。
- **启动脚本**：用户交互的入口点。负责环境检测、预检查、JVM 参数组装、应用进程启动、控制台日志输出。分为 `start.bat`（Windows 批处理）和 `start.sh`（POSIX Shell 脚本）。
- **预检查模块**：启动脚本中封装的检查逻辑集合。包含 JRE 检查、配置检查、端口检查、权限检查、磁盘检查。每个检查独立执行，失败时输出对应错误信息。
- **启动信息横幅**：应用完全启动后输出到控制台的信息块。包含访问地址列表（含回环和网络接口地址）、端口号、上下文路径、主要功能路径。

## 成功标准 *（强制）*

### 可衡量的结果

- **SC-001**：用户从解压启动包到服务成功启动并可通过浏览器访问管理界面的时间不超过 3 分钟（在操作系统已安装的必要依赖之外，无需额外安装步骤）。
- **SC-002**：启动脚本对 5 种常见启动故障（端口占用、配置文件缺失、磁盘空间不足、JRE 缺失、数据目录无写入权限）的检测覆盖率达到 100%，每种故障均输出中文错误描述和具体修复建议。
- **SC-003**：启动成功横幅中输出的网络地址必须覆盖所有活跃网络接口，包括至少 1 个回环地址（`127.0.0.1`）和全部非回环 IPv4 地址（如 `192.168.x.x`、`10.x.x.x` 等），地址数量与操作系统网络接口数量一致。
- **SC-004**：密码自动加密功能在应用启动后 5 秒内完成检测和加密（无需额外手动步骤），加密后的密码可通过标准 BCrypt 验证。
- **SC-005**：打包脚本从干净仓库到生成最终 zip/tar.gz 产物的时间不超过 5 分钟（依赖缓存有效的前提下）。
- **SC-006**：生成的启动包解压后在全新安装的操作系统（Windows 10+/Linux kernel 5.x+，无预装 Java/Maven/Node.js）上可直接运行，无需联网下载任何额外组件。
- **SC-007**：启动包大小（压缩后）不超过 150MB，解压后不超过 400MB。

## 假设

- **A1**：目标用户的操作系统为 Windows 10/11（x86-64）或 Linux（x86-64, kernel 5.x+）。macOS 和 ARM 架构在 v1 版本中不作为主要目标，但 Linux ARM 的支持框架保留扩展空间。
- **A2**：用户有权限在目标机器上监听网络端口（>=1024 的端口无需管理员/root 权限）。默认端口 8080 为普通用户可用的非特权端口。
- **A3**：JRE 运行时通过 Adoptium（Eclipse Temurin）发行版获取，该发行版允许重新分发。打包脚本从 Adoptium API 下载对应平台的 JRE tar.gz/zip 文件。
- **A4**：前端构建需要 Node.js 环境——但这仅在开发者打包阶段需要，最终分发给用户的启动包中不包含 Node.js，仅包含前端构建产物（静态 HTML/JS/CSS 文件）。
- **A5**：H2 数据库文件存储在启动包内的 `data/` 目录下，默认路径可通过 `DATA_DIR` 环境变量覆盖。启动包解压目录需有写入权限。
- **A6**：启动脚本的预检查逻辑用 Shell/PowerShell 实现，不引入额外脚本语言运行时（如 Python）。Java 层面的启动后处理（如密码加密、地址输出）用 Java 实现，作为应用启动流程的一部分。
- **A7**：打包过程将复用现有的 `mvnw`（Maven Wrapper），不会强制开发者安装特定版本的 Maven。前端构建依赖 Node.js（开发者环境已有），打包脚本检查 Node.js 可用性，若不可用则跳过前端构建并警告。
- **A8**：密码加密采用与现有项目一致的 Spring Security Crypto BCrypt 实现，加密后的密码以 `{bcrypt}` 前缀标识（与当前 application.yaml 中的格式一致）。密码加密逻辑在 Spring Boot 应用启动的最早阶段（`ApplicationContextInitializer` 或 `EnvironmentPostProcessor`）执行，在认证拦截器初始化之前完成。
- **A9**：启动脚本不提供"后台运行/服务化"功能（如注册为 Windows 服务或 Linux systemd 单元）。用户如有此类需求，可自行配置。v1 聚焦于前台交互式启动。
- **A10**：Docker 场景下的地址输出通过同一套 Java 代码实现（`ApplicationReadyEvent` 监听器），复用一体化启动的地址输出逻辑，仅在输出提示文案上区分容器环境。
