## 检查清单的用途："需求的单元测试"

**核心概念**：检查清单是**需求编写的单元测试** — 它们验证给定领域中需求的质量、清晰度和完整性。

**不是用于验证/测试**：

- ❌ 不是"验证按钮点击正确"
- ❌ 不是"测试错误处理是否正常"
- ❌ 不是"确认 API 返回 200"
- ❌ 不是检查代码/实现是否与规格匹配

**用于需求质量验证**：

- ✅ "是否所有卡片类型都定义了视觉层次需求？"（完整性）
- ✅ "'突出显示'是否用具体的尺寸/定位进行了量化？"（清晰度）
- ✅ "所有交互元素的悬停状态需求是否一致？"（一致性）
- ✅ "是否定义了键盘导航的无障碍需求？"（覆盖度）
- ✅ "规格是否定义了 logo 图片加载失败时的行为？"（边界情况）

**比喻**：如果你的规格是用英文写的代码，检查清单就是它的单元测试套件。你正在测试需求是否编写良好、完整、无歧义并准备就绪 — 而不是测试实现是否正常工作。

## 用户输入

```text
$ARGUMENTS
```

**必须**在继续之前考虑用户输入（如果不为空）。

## 预执行检查

**检查扩展钩子（检查清单生成之前）**：
- 检查项目根目录中是否存在 `.specify/extensions.yml`。
- 如果存在，读取该文件并查找 `hooks.before_checklist` 键下的条目。
- 如果 YAML 无法解析或无效，静默跳过钩子检查并正常继续。
- 过滤掉 `enabled` 显式为 `false` 的钩子。将没有 `enabled` 字段的钩子视为默认启用。
- 对于每个剩余的钩子，**不要**尝试解释或评估钩子的 `condition` 表达式：
  - 如果钩子没有 `condition` 字段，或其值为 null/空，将钩子视为可执行。
  - 如果钩子定义了非空的 `condition`，跳过该钩子，将条件评估交由 HookExecutor 实现处理。
- 从钩子命令名称构造斜杠命令时，将点号（`.`）替换为连字符（`-`）。例如，`speckit.git.commit` → `/speckit-git-commit`。
- 对于每个可执行的钩子，根据其 `optional` 标志输出以下内容：
  - **可选钩子**（`optional: true`）：
    ```
    ## 扩展钩子

    **可选预检钩子**：{extension}
    命令：`/{command}`
    描述：{description}

    提示：{prompt}
    执行方式：`/{command}`
    ```
  - **强制钩子**（`optional: false`）— **必须为每个强制钩子输出 `EXECUTE_COMMAND:`**：
    ```
    ## 扩展钩子

    **自动预检钩子**：{extension}
    正在执行：`/{command}`
    EXECUTE_COMMAND：{command}

    请等待钩子命令的结果，然后再继续执行步骤。
    ```
- 如果没有注册钩子或 `.specify/extensions.yml` 不存在，静默跳过。

## 执行步骤

1. **设置**：从仓库根目录运行 `.specify/scripts/powershell/check-prerequisites.ps1 -Json`，解析 JSON 获取 FEATURE_DIR 和 AVAILABLE_DOCS 列表。
   - 所有文件路径必须是绝对路径。
   - 对于参数中的单引号，如 "I'm Groot"，使用转义语法：例如 'I'\''m Groot'（或尽可能使用双引号："I'm Groot"）。

2. **如果存在**：加载 `.specify/memory/constitution.md` 以获取项目原则和治理约束。

3. **澄清意图（动态）**：推导最多三个初步的上下文澄清问题（没有预设目录）。它们必须：
   - 根据用户的措辞 + 从 spec/plan/tasks 提取的信号生成。
   - 仅询问会实质性改变检查清单内容的信息。
   - 如果已在 `$ARGUMENTS` 中明确，则单独跳过。
   - 优先精确而非广泛。

   生成算法：
   1. 提取信号：功能领域关键词（例如，认证、延迟、UX、API）、风险指标（"关键"、"必须"、"合规"）、相关方提示（"QA"、"审查"、"安全团队"）和明确的交付物（"无障碍"、"回滚"、"契约"）。
   2. 将信号聚集为候选聚焦领域（最多 4 个），按相关性排名。
   3. 识别可能的受众和时机（作者、审查者、QA、发布），如果未明确说明。
   4. 检测缺失维度：范围广度、深度/严谨度、风险强调、排除边界、可衡量的验收标准。
   5. 从以下原型中制定问题：
      - 范围细化（例如，"此检查清单是否应包含与 X 和 Y 的集成接触点，还是仅限于本地模块正确性？"）
      - 风险优先级（例如，"以下哪些潜在风险领域应接受强制性门禁检查？"）
      - 深度校准（例如，"这是轻量级的提交前健全性检查，还是正式的发布门禁？"）
      - 受众框架（例如，"此检查清单仅供作者使用，还是供同行在 PR 审查期间使用？"）
      - 边界排除（例如，"本轮是否应明确排除性能调优项？"）
      - 场景类缺口（例如，"未检测到恢复流程 — 回滚/部分失败路径是否在范围内？"）

   问题格式规则：
   - 如果提供选项，生成一个紧凑的表格，包含列：选项 | 候选项 | 为什么重要
   - 最多限制 A–E 选项；如果自由形式答案更清晰则省略表格。
   - 绝不让用户重复其已经说过的话。
   - 避免推测性类别（不要虚构）。如果不确定，明确询问："确认 X 是否属于范围。"

   无法交互时的默认值：
   - 深度：标准
   - 受众：审查者（PR），如果与代码相关；否则为作者。
   - 聚焦：前 2 个相关性集群。

   输出问题（标记为 Q1/Q2/Q3）。回答后：如果仍有 ≥2 个场景类（替代流程/异常/恢复/非功能领域）不清楚，可以最多再问两个有针对性的跟进问题（Q4/Q5），每个附带一行理由（例如，"未解决的恢复路径风险"）。总共不要超过五个问题。如果用户明确拒绝更多问题，则跳过升级。

4. **理解用户需求**：结合 `$ARGUMENTS` + 澄清答案：
   - 推导检查清单主题（例如，安全、审查、部署、UX）。
   - 整合用户提到的明确必须有项。
   - 将聚焦选择映射到类别框架。
   - 从 spec/plan/tasks 推断任何缺失的上下文（不要虚构）。

5. **加载功能上下文**：从 FEATURE_DIR 读取：
   - spec.md：功能需求和范围。
   - plan.md（如果存在）：技术细节、依赖关系。
   - tasks.md（如果存在）：实现任务。

   **上下文加载策略**：
   - 仅加载与当前聚焦领域相关的必要部分（避免全文倾倒）。
   - 优先将长段落概括为简洁的场景/需求要点。
   - 使用渐进式披露：仅在检测到缺口时追加检索。
   - 如果源文档很大，生成中期摘要项，而不是嵌入原始文本。

6. **生成检查清单** — 创建"需求的单元测试"：
   - 如果 `FEATURE_DIR/checklists/` 目录不存在，则创建。
   - 生成唯一的检查清单文件名：
     - 使用基于领域的简短、描述性名称（例如，`ux.md`、`api.md`、`security.md`）。
     - 格式：`[domain].md`。
   - 文件处理行为：
     - 如果文件不存在：创建新文件并从 CHK001 开始编号。
     - 如果文件存在：将新项追加到现有文件，从最后一个 CHK ID 继续（例如，如果最后一项是 CHK015，新项从 CHK016 开始）。
   - 绝不删除或替换现有检查清单内容 — 始终保留并追加。

   **核心原则 — 测试需求，而非实现**：
   每个检查清单项必须评估**需求本身**的以下方面：
   - **完整性**：是否所有必要的需求都已存在？
   - **清晰度**：需求是否无歧义且具体？
   - **一致性**：需求是否相互对齐？
   - **可衡量性**：需求是否可以客观验证？
   - **覆盖度**：是否所有场景/边界情况都已处理？

   **类别结构** — 按需求质量维度分组：
   - **需求完整性**（是否所有必要的需求都已记录？）
   - **需求清晰度**（需求是否具体且无歧义？）
   - **需求一致性**（需求是否无冲突地对齐？）
   - **验收标准质量**（成功标准是否可衡量？）
   - **场景覆盖度**（是否所有流程/情况都已处理？）
   - **边界情况覆盖度**（是否定义了边界条件？）
   - **非功能需求**（性能、安全性、无障碍等 — 是否已指定？）
   - **依赖与假设**（是否已记录并验证？）
   - **歧义与冲突**（哪些需要澄清？）

   **如何编写检查清单项 — "需求的单元测试"**：

   ❌ **错误**（测试实现）：
   - "验证着陆页显示 3 张剧集卡片"
   - "测试桌面端悬停状态是否正常"
   - "确认点击 logo 导航回首页"

   ✅ **正确**（测试需求质量）：
   - "是否指定了精选剧集的确切数量和布局？" [完整性]
   - "'突出显示'是否用具体的尺寸/定位进行了量化？" [清晰度]
   - "所有交互元素的悬停状态需求是否一致？" [一致性]
   - "是否所有交互 UI 都定义了键盘导航需求？" [覆盖度]
   - "是否指定了 logo 图片加载失败时的回退行为？" [边界情况]
   - "是否定义了异步剧集数据的加载状态？" [完整性]
   - "规格是否定义了竞争 UI 元素的视觉层次？" [清晰度]

   **项结构**：
   每个项应遵循以下模式：
   - 询问需求质量的问题格式。
   - 聚焦于规格/计划中已写（或未写）的内容。
   - 在方括号中包含质量维度 [完整性/清晰度/一致性/等]。
   - 在检查现有需求时引用规格章节 `[Spec §X.Y]`。
   - 在检查缺失需求时使用 `[缺口]` 标记。

   **按质量维度的示例**：

   完整性：
   - "是否所有 API 失败模式都定义了错误处理需求？[缺口]"
   - "是否所有交互元素都指定了无障碍需求？[完整性]"
   - "是否为响应式布局定义了移动端断点需求？[缺口]"

   清晰度：
   - "'快速加载'是否用具体的时间阈值进行了量化？[清晰度, Spec §NFR-2]"
   - "'相关剧集'的选择标准是否明确定义？[清晰度, Spec §FR-5]"
   - "'突出'是否用可衡量的视觉属性进行了定义？[歧义, Spec §FR-4]"

   一致性：
   - "导航需求是否在所有页面之间保持一致？[一致性, Spec §FR-10]"
   - "着陆页和详情页之间的卡片组件需求是否一致？[一致性]"

   覆盖度：
   - "是否为零态场景（无剧集）定义了需求？[覆盖度, 边界情况]"
   - "是否处理了并发用户交互场景？[覆盖度, 缺口]"
   - "是否为部分数据加载失败指定了需求？[覆盖度, 异常流程]"

   可衡量性：
   - "视觉层次需求是否可衡量/可测试？[验收标准, Spec §FR-1]"
   - "'平衡的视觉权重'是否可以客观验证？[可衡量性, Spec §FR-2]"

   **场景分类与覆盖度**（需求质量聚焦）：
   - 检查是否存在以下需求：主流程、替代流程、异常/错误、恢复、非功能场景。
   - 对于每个场景类，询问："[场景类型]需求是否完整、清晰且一致？"
   - 如果场景类缺失："[场景类型]需求是有意排除还是缺失？[缺口]"
   - 当发生状态变更时包含韧性/回滚："是否为迁移失败定义了回滚需求？[缺口]"

   **可追溯性需求**：
   - 最低要求：≥80% 的项必须包含至少一个可追溯性引用。
   - 每个项应引用：规格章节 `[Spec §X.Y]`，或使用标记：`[缺口]`、`[歧义]`、`[冲突]`、`[假设]`。
   - 如果没有 ID 体系存在："是否建立了需求和验收标准的 ID 方案？[可追溯性]"

   **发现和解决问题**（需求质量问题）：
   提出关于需求本身的问题：
   - 歧义："'快速'一词是否用具体指标量化？[歧义, Spec §NFR-1]"
   - 冲突："导航需求是否在 §FR-10 和 §FR-10a 之间存在冲突？[冲突]"
   - 假设："'播客 API 始终可用'的假设是否已验证？[假设]"
   - 依赖："外部播客 API 需求是否已记录？[依赖, 缺口]"
   - 缺失定义："'视觉层次'是否用可衡量标准定义？[缺口]"

   **内容整合**：
   - 软上限：如果原始候选项超过 40 个，按风险/影响优先级排序。
   - 合并检查同一需求方面的近似重复项。
   - 如果超过 5 个低影响的边界情况，创建一个项："边界情况 X、Y、Z 是否在需求中处理？[覆盖度]"

   **🚫 绝对禁止** — 这些会使其成为实现测试，而不是需求测试：
   - ❌ 任何以"验证"、"测试"、"确认"、"检查" + 实现行为开头的项。
   - ❌ 引用代码执行、用户操作、系统行为。
   - ❌ "显示正确"、"工作正常"、"功能如预期"。
   - ❌ "点击"、"导航"、"渲染"、"加载"、"执行"。
   - ❌ 测试用例、测试计划、QA 流程。
   - ❌ 实现细节（框架、API、算法）。

   **✅ 必需模式** — 这些测试需求质量：
   - ✅ "是否[需求类型]为[场景]定义/指定/记录了？"
   - ✅ "是否[模糊术语]用具体标准进行了量化/澄清？"
   - ✅ "需求在[章节 A]和[章节 B]之间是否一致？"
   - ✅ "是否[需求]可以客观衡量/验证？"
   - ✅ "需求中是否处理了[边界情况/场景]？"
   - ✅ "规格是否定义了[缺失的方面]？"

7. **结构参考**：按照 `.specify/templates/checklist-template.md` 中的规范模板生成检查清单，用于标题、元信息部分、类别标题和 ID 格式。如果模板不可用，使用：H1 标题、目的/创建元信息行、`##` 类别部分包含 `- [ ] CHK### <需求项>` 行，全局递增 ID 以 CHK001 开始。

8. **报告**：输出检查清单文件的完整路径、项数量，并摘要说明本次运行是创建了新文件还是追加到现有文件。总结：
   - 选择的聚焦领域。
   - 深度级别。
   - 角色/时机。
   - 任何已纳入的用户指定必须有项。

**重要**：每次 `/speckit-checklist` 命令调用使用简短、描述性的检查清单文件名，并创建新文件或追加到现有文件。这允许：

- 不同类型的多个检查清单（例如，`ux.md`、`test.md`、`security.md`）。
- 简单、易记的文件名，表明检查清单用途。
- 在 `checklists/` 文件夹中轻松识别和导航。

为避免混乱，使用描述性类型，并在完成后清理过时的检查清单。

## 检查清单类型与示例项

**UX 需求质量：** `ux.md`

示例项（测试需求，而非实现）：

- "视觉层次需求是否用可衡量标准定义？[清晰度, Spec §FR-1]"
- "UI 元素的数量和位置是否明确指定？[完整性, Spec §FR-1]"
- "交互状态需求（悬停、聚焦、激活）是否一致定义？[一致性]"
- "是否所有交互元素都指定了无障碍需求？[覆盖度, 缺口]"
- "图片加载失败时的回退行为是否定义？[边界情况, 缺口]"
- "'突出显示'是否可以客观衡量？[可衡量性, Spec §FR-4]"

**API 需求质量：** `api.md`

示例项：

- "是否所有失败场景都指定了错误响应格式？[完整性]"
- "速率限制需求是否用具体阈值量化？[清晰度]"
- "所有接口的身份验证需求是否一致？[一致性]"
- "是否为外部依赖定义了重试/超时需求？[覆盖度, 缺口]"
- "版本控制策略是否在需求中记录？[缺口]"

**性能需求质量：** `performance.md`

示例项：

- "性能需求是否用具体指标量化？[清晰度]"
- "是否所有关键用户旅程都定义了性能目标？[覆盖度]"
- "是否指定了不同负载条件下的性能需求？[完整性]"
- "性能需求是否可以客观衡量？[可衡量性]"
- "是否为高负载场景定义了降级需求？[边界情况, 缺口]"

**安全需求质量：** `security.md`

示例项：

- "是否所有受保护资源都指定了身份验证需求？[覆盖度]"
- "是否为敏感信息定义了数据保护需求？[完整性]"
- "威胁模型是否已记录且需求与之对齐？[可追溯性]"
- "安全需求是否与合规义务一致？[一致性]"
- "是否定义了安全失败/泄露的响应需求？[缺口, 异常流程]"

## 反面示例：什么不该做

**❌ 错误 — 这些测试实现，而非需求：**

```markdown
- [ ] CHK001 — 验证着陆页显示 3 张剧集卡片 [Spec §FR-001]
- [ ] CHK002 — 测试桌面端悬停状态正常工作 [Spec §FR-003]
- [ ] CHK003 — 确认点击 logo 导航到首页 [Spec §FR-010]
- [ ] CHK004 — 检查相关剧集部分显示 3-5 项 [Spec §FR-005]
```

## 执行后检查

**检查扩展钩子（检查清单生成之后）**：
检查项目根目录中是否存在 `.specify/extensions.yml`。
- 如果存在，读取该文件并查找 `hooks.after_checklist` 键下的条目。
- 如果 YAML 无法解析或无效，静默跳过钩子检查并正常继续。
- 过滤掉 `enabled` 显式为 `false` 的钩子。将没有 `enabled` 字段的钩子视为默认启用。
- 对于每个剩余的钩子，**不要**尝试解释或评估钩子的 `condition` 表达式：
  - 如果钩子没有 `condition` 字段，或其值为 null/空，将钩子视为可执行。
  - 如果钩子定义了非空的 `condition`，跳过该钩子，将条件评估交由 HookExecutor 实现处理。
- 从钩子命令名称构造斜杠命令时，将点号（`.`）替换为连字符（`-`）。例如，`speckit.git.commit` → `/speckit-git-commit`。
- 对于每个可执行的钩子，根据其 `optional` 标志输出以下内容：
  - **可选钩子**（`optional: true`）：
    ```
    ## 扩展钩子

    **可选钩子**：{extension}
    命令：`/{command}`
    描述：{description}

    提示：{prompt}
    执行方式：`/{command}`
    ```
  - **强制钩子**（`optional: false`）— **必须为每个强制钩子输出 `EXECUTE_COMMAND:`**：
    ```
    ## 扩展钩子

    **自动钩子**：{extension}
    正在执行：`/{command}`
    EXECUTE_COMMAND：{command}
    ```
- 如果没有注册钩子或 `.specify/extensions.yml` 不存在，静默跳过。
